# 青数 × QingShuClaw 通俗版架构图说明

配套图文件：
`docs/qingshu-enterprise-ai-architecture-simple.excalidraw`

这是一版给非技术同学看的说明稿，重点不是系统细节，而是回答两个问题：

1. 企业的数据是怎么被管理起来的？
2. 这些被管理好的数据，最后给业务带来了什么价值？

## 1. 一句话先讲清楚

青数负责把企业里分散、复杂、不统一的数据整理好、讲清楚、管起来。

QingShuClaw 负责把这些被整理好的能力，真正送到业务人员面前，让大家可以直接提问题、做分析、拿结果。

## 2. 数据管理逻辑，通俗理解

可以把它理解成 5 步：

1. 先把数据接进来  
   企业里原本有很多数据，可能来自数据库、业务系统、数仓、LBS 系统、字典系统。第一步就是把这些数据统一接入。

2. 再把数据整理成“大家都能理解的一套口径”  
   这里会形成数据模型、数据集、元数据、术语、标签。  
   通俗说，就是把“这份数据到底代表什么、该怎么理解、哪些字段是什么意思”都定义清楚。

3. 再把数据包装成可调用的能力  
   不只是把表放在那里，而是把它们做成 Tool、Skill、Agent。  
   这样 AI 才不是直接乱查数据，而是按规则去使用被治理过的数据能力。
   
   更重要的是，每个数据域里和 Tool 绑定的 SQL，并不是一次性脚本，而会逐步沉淀成这个数据域的“知识库”。  
   这些知识沉淀下来之后，就可以反过来支撑后续的 AI 自动生成 SQL、自动规划 Tool、自动组装 Skill，进一步为自动分析和自动执行打基础。

4. 再把这些能力发布出来  
   青数统一管理版本、权限、描述、发布方式，再把这些能力同步给 QingShuClaw。

5. 最后让业务真正用起来  
   用户在 QingShuClaw 里直接提需求，比如做分析、出报告、做 PPT。  
   用的过程中，新的反馈又会回到平台，继续优化数据口径和能力设计。

## 3. 为什么“Tool 绑定的 SQL”会越用越值钱

这部分可以用一句很通俗的话来理解：

企业不是每次都从零开始写 SQL，而是在不断积累“这个业务领域里，什么 SQL 是好用的、可信的、可复用的”。

所以，当一个数据域里越来越多 Tool 被建设出来后，背后的 SQL、字段口径、筛选条件、分析规则，就会慢慢沉淀成这个数据域的经验库。

这个经验库的价值，不只是今天能支撑人工配置 Tool，更重要的是能支撑未来的 AI 自动化能力，比如：

- 自动生成 SQL
- 自动规划应该用哪些 Tool
- 自动生成 Tool
- 自动组装成 Skill
- 进一步走向自动分析和自动执行

所以这不是一套“只解决眼前使用问题”的架构，它同时也在为未来更强的 AI 能力打地基。

## 4. 产品价值，通俗理解

这套体系最核心的价值有 4 个：

- 统一口径  
  不同团队、不同岗位，不再各说各话，用的是同一套数据定义。

- 降低门槛  
  以前很多数据分析必须找技术或数据同学，现在业务人员也可以直接用。

- 能力可复制  
  一个沉淀好的 Tool、Skill、Agent，不是一次性成果，而是可以反复复用的能力资产。

- 持续优化  
  真实使用会不断带来反馈，平台就能越来越懂业务，能力也会越来越稳定。

- 为未来自动化打基础  
  当越来越多高质量 SQL、Tool、Skill 被沉淀下来，平台就不只是“人工配置能力”，而会逐步进化到“AI 辅助生成能力”，让自动分析成为可能。

## 5. 青数和 QingShuClaw 为什么要配合

如果只有青数，没有 QingShuClaw，那么很多能力会停留在“后台治理得很好”，但前线业务不一定真的会用。

如果只有 QingShuClaw，没有青数，那么前台虽然能对话，但底层数据口径、权限、规则不统一，结果很容易不稳定。

所以二者缺一不可：

- 青数像“能力工厂”
- QingShuClaw 像“业务使用入口”

前者负责把能力造好，后者负责把能力用好。

## 6. 最适合对外讲的一句话

青数 + QingShuClaw，不只是“把数据接给 AI”，而是把企业的数据先治理成标准能力，再把这些能力送到业务现场，真正形成企业级的数据 AI 闭环。

再往后看，这套体系还有一个更大的价值：它会不断沉淀各个数据域的 SQL 与能力资产，最终为 AI 自动生成 SQL、自动生成 Tool、自动规划 Skill 提供天然知识底座。
