# 青数飞书登录认证

## 1. 文档目标

本文用于完整梳理当前 `QingShuClaw + qtb-data-platform` 的登录认证标准流程，覆盖以下内容：

- 平台用户、平台权限、桌面会话、Web 会话之间的关系
- `QingShuClaw` 的账号密码登录、飞书扫码登录、浏览器飞书授权登录
- 青数 Web 的账号密码登录、飞书网页登录
- `QingShuClaw <-> 青数 Web` 双向免登
- token、refresh、退出登录、登录失效的统一规则
- 飞书开放平台应用申请、回调配置、开发环境与生产环境配置方式
- 后续如何 1:1 复制到其他应用，或按同样模式实现其他三方登录

本文以当前项目已落地的代码与联调结论为准，并补充飞书开放平台接入时必须关注的配置要求。

## 2. 适用范围

本文适用于以下两类场景：

1. 为另一个桌面端或 Web 应用复制一套与青数一致的飞书登录认证机制。
2. 以飞书为模板，实现钉钉、企微、Google、GitHub 等其他第三方登录。

如果只是查看当前 `QingShuClaw` 与青数的认证现状，可同时配合阅读：

- `docs/qingshu-auth-bridge-overview.md`

## 3. 设计原则

### 3.1 统一用户与权限，不统一所有 token 形态

当前方案统一的是：

- 用户身份语义
- 权限判定入口
- 跨端免登能力

当前方案不强求一开始就统一：

- Web token 的全部 claims
- 桌面 token 的全部 claims
- 扫码 token、桥接 token、网页登录 token 的内部格式

这是当前阶段更符合 `KISS / YAGNI` 的做法：

- 能复用既有平台登录与鉴权体系
- 能快速支持桌面端完整能力
- 不会一次性把所有历史链路都推翻重做

### 3.2 桌面端优先使用“桌面专用会话”

对于桌面应用，推荐以“桌面双 token 会话”为标准：

- `accessToken`
- `refreshToken`

而不是直接复用 Web 单 token。

原因：

- 桌面端有本地持久化需求
- 桌面端要支持 refresh
- 桌面端要支持桥接免登、模型调用、额度刷新、头像摘要等专用能力

### 3.3 跨端免登使用一次性桥接票据，不直传业务 token

跨端登录不推荐在 URL 上传真实 token，而是推荐：

1. 服务端签发短时一次性 `bridge code`
2. 目标端拿 `bridge code` 到服务端换取本端会话
3. 本端自行落地 token

这样更安全，也更利于把 Web 与桌面端解耦。

## 4. 当前统一口径

### 4.1 用户体系

当前统一身份口径为 `qtb-data-platform` 平台用户。

这意味着：

- `QingShuClaw` 不维护独立账号域
- 账密登录和飞书登录最终都映射到平台用户
- Web 与桌面端访问权限时，识别的都是同一个平台用户

### 4.2 权限体系

当前统一权限口径为 `qtb-data-platform` 平台权限体系。

这意味着：

- Web 仍走平台现有当前用户与权限接口
- `QingShuClaw` 相关接口仍通过平台用户身份识别用户
- 套餐、额度、模型、头像、服务端模型调用都由同一平台身份驱动

### 4.3 三类会话

当前系统同时存在三类会话：

1. Web 登录态
   - 写入 `localStorage[SUPERSONIC_TOKEN]`
   - 用于青数 Web 页面访问
2. 桌面端登录态
   - 落地在 `QingShuClaw` 主进程本地 `auth_tokens`
   - 保存 `accessToken + refreshToken`
3. 桥接票据
   - 仅用于跨端免登
   - 一次性、短时有效
   - 不直接用于业务接口访问

## 5. 总体架构

### 5.1 参与方

- `QingShuClaw`
  - 桌面端应用
  - 负责登录 UI、token 落地、用户态恢复、额度/模型刷新
- `qtb-data-platform`
  - 平台后端
  - 负责账号密码校验、飞书授权登录、扫码会话、桥接票据、桌面专用接口
- 青数 Web
  - 平台前端
  - 负责 Web 登录页、Web 回调页、Web bridge 页
- 飞书开放平台
  - 负责 OAuth / 授权码 / 扫码授权能力

### 5.2 推荐架构分层

推荐把认证能力按下面三层设计：

1. 平台身份层
   - 用户体系
   - 权限体系
   - 第三方身份绑定关系
2. 会话签发层
   - Web token
   - 桌面 token
   - refresh token
   - bridge ticket
3. 应用接入层
   - Web 登录页 / 回调页 / bridge 页
   - 桌面登录面板 / 协议回调 / 本地持久化

## 6. 飞书开放平台申请与配置

这一部分是后续复制到其他应用时最容易漏项的地方。

### 6.1 应用申请

推荐在飞书开放平台创建一个自建应用，用于企业内部登录认证。

建议至少准备以下信息：

- 应用名称
- 应用图标
- 应用描述
- 应用所属企业 / 团队
- 登录回调域名或回调地址

一般需要从飞书开放平台拿到：

- `App ID`
- `App Secret`

这两个值是服务端向飞书发起授权 / 换 token 的基础凭证。

### 6.2 回调 URL 的标准配置

当前青数登录链路使用的飞书回调路由标准形态为：

```text
<QTB_PUBLIC_BASE_URL>/api/auth/feishu/callback/router
```

例如：

```text
http://localhost:9080/api/auth/feishu/callback/router
http://192.168.214.120:9080/api/auth/feishu/callback/router
https://auth.example.com/api/auth/feishu/callback/router
```

这里的关键点只有一个：

- 飞书开放平台中配置的 `redirect_uri`
- 实际发起授权时请求 URL 中带上的 `redirect_uri`

两者必须严格一致。

如果不一致，飞书通常会返回类似错误：

- `redirect_uri` 不匹配
- `重定向 URL 有误`
- 错误码如 `20029`

### 6.3 开发环境与生产环境的推荐做法

推荐把“飞书回调基地址”收敛为服务端一个统一配置，例如：

- `publicBaseUrl`
- `publicAuthBaseUrl`
- 或其他等价的“对外可访问地址”

然后由服务端统一推导：

```text
callbackUrl = <publicBaseUrl>/api/auth/feishu/callback/router
```

这样做的好处：

- 开发环境切生产环境时，只改一个对外基地址
- Web 登录、扫码登录、回调页都能统一使用同一个来源
- 不需要在多个页面、多个客户端里到处写死绝对路径

推荐配置策略：

1. 本机调试，只在本机浏览器里完成授权：
   - 可用 `http://localhost:9080`
2. 手机飞书扫码调试：
   - 必须改成手机能访问到的局域网 IP 或公网域名
3. 生产环境：
   - 必须使用稳定公网域名，优先 `https`

### 6.4 localhost 与局域网 IP 的区别

这是飞书扫码接入时最常见的坑。

`localhost` 只代表“当前机器自己”。

因此：

- 如果飞书授权回调地址是 `http://localhost:9080/...`
- 你在手机飞书里扫码
- 手机并不能把 `localhost` 识别成你电脑上的服务

这会导致：

- 手机扫码后打不开回调页
- 或回调阶段失败
- 或二维码始终无法完成绑定

所以：

- 本机浏览器授权，可以用 `localhost`
- 手机扫码登录，不应使用 `localhost`
- 手机扫码时，应改用 `http://<局域网IP>:9080/...` 或公网地址

### 6.5 飞书开放平台回调配置检查清单

每次配置新环境，都应检查以下项目：

1. 飞书开放平台上的 `redirect_uri` 配置是否与实际授权请求一致。
2. 服务端对外基地址是否已改成当前环境真实可访问地址。
3. 手机扫码时，回调地址是否可从手机访问。
4. 飞书应用是否已保存并发布到当前可用版本。
5. 当前使用的 `App ID / App Secret` 是否与该环境匹配。

### 6.6 推荐的飞书授权页 / 扫码页形态

当前实践中，可把飞书授权 URL 理解为两类：

1. 浏览器授权页
   - 由服务端 `GET /api/auth/feishu/authorize` 统一生成
2. 扫码页 / 扫码二维码内容
   - 由服务端扫码会话接口统一生成

如果你需要自己理解或拼接扫码页，常见外层页面形态类似：

```text
https://open.feishu.cn/open-apis/authen/v1/index?app_id=<APP_ID>&redirect_uri=<URL_ENCODED_REDIRECT_URI>&state=<STATE>
```

但在工程实践里，推荐仍由服务端统一生成最终 `authorizeUrl` 或二维码内容，客户端不要自己拼授权参数。

原因：

- 避免客户端和服务端各自维护一套授权参数
- 避免 `state`、回调地址、环境切换产生分叉
- 更利于后续复制到其他三方登录

## 7. 标准登录流程

## 7.1 Web 账号密码登录

### 流程

```text
Web 登录页
-> 输入用户名、密码
-> POST /api/auth/user/login
-> 返回平台 token
-> localStorage[SUPERSONIC_TOKEN]
-> 拉当前用户与权限
-> 跳转首页
```

### 特点

- 用户源头是平台用户
- 属于 Web 会话
- 不涉及桌面 token

## 7.2 Web 飞书网页登录

### 流程

```text
Web 登录页
-> 跳转飞书授权页
-> 飞书回调到 /api/auth/feishu/callback/router
-> Web Callback 页拿到 code
-> POST /api/auth/feishu/login
-> 写入 SUPERSONIC_TOKEN
-> 拉当前用户与权限
-> 跳转首页
```

### 特点

- 属于标准 Web OAuth 登录
- 最终落地的是 Web token
- 当前平台已有现成登录页与回调页

## 7.3 QingShuClaw 账号密码登录

### 标准流程

```text
QingShuClaw 登录面板
-> 输入用户名、密码
-> POST /api/qingshu-claw/auth/password-login
-> 返回桌面 accessToken + refreshToken
-> 主进程保存 auth_tokens
-> 拉用户 / 套餐额度 / 头像摘要 / 可用模型
-> 进入登录态
```

### 说明

这条链路是当前桌面端推荐标准。

它的优点是：

- 不依赖浏览器
- 不依赖 deep link
- 能直接拿到桌面双 token
- refresh 机制完整

### 当前客户端实现要点

- 用户输入的密码会按现有平台规则先加密
- 桌面端落地的是主进程本地双 token
- renderer 不单独持久化业务 token

## 7.4 QingShuClaw 默认飞书扫码登录

### 标准流程

```text
QingShuClaw 登录面板打开
-> POST /api/datachat/qingshu/auth/scan/session
-> 返回 scanSessionId + authorizeUrl + qrCodeContent + expiredAt
-> 登录面板直接展示二维码
-> 客户端轮询 GET /api/datachat/qingshu/auth/scan/session/{scanSessionId}
-> 用户在飞书扫码并确认
-> 服务端返回桌面 accessToken + refreshToken
-> 主进程保存 auth_tokens
-> 拉用户 / 套餐额度 / 头像摘要 / 可用模型
-> 进入登录态
```

### 特点

- 当前桌面端默认登录方式
- 不需要系统浏览器
- 用户停留在桌面端面板内完成登录
- 扫码健康统计与治理页使用同一后端数据源

### 推荐性

如果目标三方平台支持扫码授权或设备码流程，这种模式应作为桌面端首选。

原因：

- 用户体验更自然
- 不依赖系统浏览器回跳
- 更适合桌面应用
- 会话模型更容易统一成桌面专用会话

## 7.5 QingShuClaw 手动模式“打开扫码页”

### 标准流程

```text
QingShuClaw 手动登录
-> 用户点击“打开扫码页”
-> POST /api/datachat/qingshu/auth/scan/session
-> 打开应用内扫码窗口 /login/desktop-scan?scanSessionId=...
-> 页面内渲染同一个扫码会话二维码
-> 客户端继续轮询 scanSessionId
-> 用户扫码确认
-> 返回桌面 accessToken + refreshToken
-> 进入登录态
```

### 特点

- 本质仍是扫码会话
- 只是承载容器从登录面板换成应用内扫码页
- 仍不建议退回到“系统浏览器 + token 回跳”作为主方案

## 7.6 QingShuClaw 浏览器飞书授权登录

### 当前兼容流程

```text
QingShuClaw 手动登录
-> 点击“在浏览器中打开飞书登录”
-> GET /api/auth/feishu/authorize
-> 系统浏览器打开飞书授权页
-> 授权成功后通过 lobsterai://auth/callback?... 回到桌面端
-> 主进程暂存 code
-> renderer 调用 POST /api/auth/feishu/login
-> 进入登录态
```

### 说明

这条链路当前仍保留，主要作用是：

- 作为扫码失败时的 fallback
- 兼容已有网页登录式授权能力

但如果要为新的桌面应用设计“标准方案”，不建议把它作为主路径。

原因：

- 它天然更像 Web OAuth 流程
- 浏览器回跳依赖更强
- 会话语义通常更难天然对齐到桌面专用双 token

更推荐的桌面标准方案仍然是：

- 账号密码登录
- 默认扫码登录
- 或独立的桌面授权换票接口

## 8. 双向免登流程

## 8.1 QingShuClaw -> 青数 Web

### 标准流程

```text
QingShuClaw 用户菜单
-> POST /api/qingshu-claw/auth/bridge/tickets { target: web }
-> 返回 bridge code
-> 打开 /login/bridge?code=...
-> Web POST /api/qingshu-claw/auth/bridge/exchange
-> 返回 Web token
-> 写入 SUPERSONIC_TOKEN
-> 拉 Web 当前用户与权限
-> 跳转目标页
```

### 特点

- URL 不直接传 token
- 服务端按 Web 目标端重新签发会话

## 8.2 青数 Web -> QingShuClaw

### 标准流程

```text
Web 头像菜单
-> POST /api/qingshu-claw/auth/bridge/tickets { target: desktop }
-> 返回 bridge code
-> 跳 lobsterai://auth/bridge?code=...
-> 桌面端主进程拿到 code
-> POST /api/qingshu-claw/auth/bridge/exchange
-> 返回桌面 accessToken + refreshToken
-> 主进程保存 auth_tokens
-> 拉用户 / 套餐额度 / 头像摘要 / 可用模型
-> 进入登录态
```

### 特点

- Web 与桌面端真正使用的是各自的目标端会话
- 不是简单共享同一个 token

## 9. Token、refresh、退出与失效机制

## 9.1 本地唯一真相

当前桌面端本地持久化唯一真相是主进程 kv store 中的：

- `auth_tokens`

内容包括：

- `accessToken`
- `refreshToken`

## 9.2 refresh 规则

当前桌面端规则如下：

1. 优先使用本地 `accessToken`
2. 若请求鉴权失败，尝试 refresh
3. 若 refresh 成功：
   - 更新本地 `accessToken + refreshToken`
   - 保持当前登录态
4. 若 refresh 失败：
   - 统一判定为会话失效
   - 执行物理清 token
   - 用户必须重新登录

## 9.3 手动退出规则

手动退出当前是“全清”。

退出时应清理：

- 本地 `auth_tokens`
- pending callback
- pending bridge code
- 授权弹窗
- renderer 登录态
- 服务器模型缓存
- 飞书扫码会话缓存

这意味着：

- 手动退出后，下次启动不会拿旧 refresh token 自动续期
- 登录失效和手动退出最终都收敛到统一清理入口

## 9.4 无 token 时的能力边界

没有有效 token 时，下面这些平台能力不可用：

- 当前用户信息
- 套餐 / 额度
- 头像摘要
- 平台可用模型
- 服务端代理模型调用
- 创建 bridge ticket

但本地自定义模型能力仍可继续使用。

## 10. 飞书扫码健康与治理对齐

当前桌面端默认扫码登录已与青数的扫码健康治理口径对齐。

核心原则是：

- 桌面端扫码不新建第二套扫码协议
- 继续复用平台后端的扫码会话服务
- 创建 session、回调处理、状态轮询、审计落库都走同一条后端链路

这意味着治理页可无差别看到：

- 成功数
- 失败数
- 待处理数
- 最近扫码会话
- 错误码
- 匹配来源
- 回调诊断

如果后续复制到其他应用，也推荐保持这一原则：

- “业务应用只是扫码前端壳”
- “扫码协议、状态机、审计、诊断都沉到平台认证层”

## 12.2 qtb-data-platform

当前配套后端实现重点在：

- 平台认证域
- 飞书授权登录
- 青数扫码会话
- QingShuClaw 专用接口
- bridge ticket 签发与换票

如果后续在其他应用复制，建议仍把这些能力放在平台统一认证域，而不是让每个应用各自维护一套。

### 13.1 服务端

### 13.2 Web 端

### 13.3 桌面端

## 14. 用同样方式实现其他三方登录的标准模板

这部分是本文最重要的可复制模板。

如果将来要接入其他三方登录，不要直接“照飞书页面硬写”，而应复制下面这套通用结构。

## 14.1 平台认证层抽象

每个第三方登录都建议拆成 5 个能力：

1. `authorizeUrl` 生成
2. `callback code` 接收
3. `code -> 平台用户` 交换
4. `平台用户 -> 目标端会话` 签发
5. `refresh / logout / bridge` 管理

推荐后端抽象为：

- `ThirdPartyIdentityService`
- `ProviderAuthorizeService`
- `ProviderCodeExchangeService`
- `PlatformSessionService`
- `BridgeTicketService`

## 14.2 应用层抽象

每个应用只做 4 件事：

1. 发起登录
2. 接收回调或轮询扫码状态
3. 保存本端 token
4. 登录成功后拉本端需要的用户态摘要

不要让应用自己负责：

- 直接和第三方用户体系做强绑定
- 自己维护一套权限表
- 自己生成 token
- 自己拼装跨端免登 token

## 14.3 推荐的通用接口模板

如果未来接 GitHub、Google、企微、钉钉，都可按下面模板展开：

### Web 登录

```text
GET  /api/auth/<provider>/authorize
POST /api/auth/<provider>/login
GET  /api/auth/<provider>/callback/router
```

### 桌面扫码或设备码登录

```text
POST /api/<app>/auth/<provider>/scan/session
GET  /api/<app>/auth/<provider>/scan/session/{id}
POST /api/<app>/auth/<provider>/refresh
```

### 桌面账号密码登录

```text
POST /api/<app>/auth/password-login
```

### 双向免登

```text
POST /api/<app>/auth/bridge/tickets
POST /api/<app>/auth/bridge/exchange
POST /api/<app>/auth/bridge/refresh
```

## 14.4 复制到其他三方登录时的建议顺序

推荐顺序如下：

1. 先打通平台 Web 登录
2. 再补桌面专用会话签发
3. 再补桌面扫码 / 设备码登录
4. 最后补 bridge 免登

原因：

- 先把身份识别打通，复杂度最低
- 再解决桌面端 token 生命周期
- 最后再做跨端体验层能力

## 15. 联调与验收清单

每次新接一个应用或新接一个三方登录，都应至少验证以下项目。

### 15.1 登录能力

- Web 账密登录成功
- Web 第三方网页登录成功
- 桌面账密登录成功
- 桌面扫码登录成功
- 桌面 fallback 登录成功

### 15.2 会话能力

- `accessToken` 可用
- `refreshToken` 可自动续签
- refresh 失败后会全清并要求重登
- 手动退出后不会自动恢复旧会话

### 15.3 跨端能力

- 桌面 -> Web 免登成功
- Web -> 桌面免登成功
- bridge code 重复消费失败
- bridge code 过期失败

### 15.4 用户与权限一致性

- 同一账号在 Web 和桌面端识别为同一个平台用户
- 同一账号在 Web 和桌面端命中同一套权限
- 同一账号的额度、套餐、头像、模型一致

### 15.5 飞书配置一致性

- 实际 `redirect_uri` 与开放平台配置一致
- 手机扫码时回调域名可访问
- 不同环境切换后仅需改 `publicBaseUrl`

## 16. 常见问题与排障

## 16.1 扫码后提示“重定向 URL 有误”

优先检查：

1. 飞书开放平台配置的回调地址是否与当前请求中的 `redirect_uri` 完全一致。
2. 你是否把 `localhost` 改成了局域网 IP，但飞书后台没有同步修改。
3. 当前环境是否切到了另一个域名，但仍在使用旧环境的飞书应用。

## 16.2 桌面端一直停留在“正在生成扫码二维码”

优先检查：

1. `/api/datachat/qingshu/auth/scan/session` 是否真的成功返回了 `scanSessionId`
2. 返回中是否带有 `qrCodeContent` 或 `authorizeUrl`
3. 服务端当前回调地址是否配置正确
4. 认证服务是否能正常访问飞书开放平台

## 16.3 手机扫码没有反应，但本机浏览器授权可以

大概率是：

- 回调地址使用了 `localhost`

处理方法：

- 换成局域网 IP 或公网域名
- 并同步更新飞书开放平台中的回调配置

## 16.4 手动退出后又自动登录

标准行为不应如此。

若出现此问题，优先检查：

- 是否真正删除了本地 `auth_tokens`
- 是否还有其他旧 token 存储点
- 是否 refresh 逻辑在退出后仍被错误触发

## 17. 推荐结论

如果后续要把这套方案复制到其他应用，推荐直接照下面的标准做：

1. 用户体系和权限体系继续统一到平台。
2. Web 与桌面端各自签发适合自己的会话。
3. 桌面端优先使用“扫码会话”或“桌面专用授权换票”，不要默认走浏览器 OAuth 回跳。
4. 免登统一使用一次性 bridge ticket。
5. 回调 URL 统一由服务端公共基地址推导，避免多处写死。

如果后续要把飞书模板复制成“其他第三方登录模板”，也推荐：

1. 保持平台认证层抽象不变
2. 只替换第三方授权页、code 换身份、扫码会话细节
3. 不重写桌面 token、refresh、bridge、退出清理等公共机制

这样做最符合当前项目的 `KISS / DRY / SOLID`：

- `KISS`
  - 统一一个平台用户口径
- `DRY`
  - refresh、bridge、logout 这些能力复用
- `SOLID`
  - 第三方登录只是身份来源扩展，不侵入业务层
