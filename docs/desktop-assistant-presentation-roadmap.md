# QingShuClaw 桌面助手演示稿路线图

## 当前已完成：B1 保守版演示稿生成

当前实现采用“原始页面不改写，额外生成演示稿”的方式：

- 从现有 `guide session` 生成 `PresentationDeck`
- 优先解析 assistant 消息中的显式分幕定义
- 对本地 `html` 目标，额外读取 html 内容并保守抽取 section/heading 作为 scene
- 在应用内生成固定 `srcDoc` 演示稿 html
- 每幕播报完成后自动推进到下一幕
- 保留暂停、继续、上一幕、下一幕、停止

在此基础上，当前又补了一层保守增强：

- 本地 `html` 拆幕时会额外识别列表、卡片标题、按钮文案、数值型指标
- 演示稿 runtime 支持 `showcase` 卡片幕与 `steps` 步骤卡片幕
- 幕内卡片使用本地 CSS 渐进出现，不依赖主进程高频驱动
- 已支持保守版问答跳幕：用户提问命中明确 scene 关键词时，会附加跳到对应幕，但不会拦截正常提问提交
- 已支持显式讲解跳幕命令：`重讲这一幕`、`回到第一页`、`跳到总结`
- 已支持讲解播报模式两档：`摘要播报` / `详细讲解`

本阶段坚持以下约束：

- 不改现有 `voice / wakeInput / tts` 公共语义
- 不改现有默认配置行为
- 不把现有手动 STT、后台唤醒、follow-up dictation 变成新模块必经节点

## B2 后续规划

### 目标

把当前“固定模板演示稿”升级为“更像 PPT 的自动演示页”，但仍保持桌面助手链路与现有语音主链路解耦。

### B2-1：HTML 自动重构为演示布局

目标：

- 不再只做固定 scene 模板
- 从原始 html 抽取视觉块、标题块、图表块、按钮区、数据卡片区
- 生成更强表达力的演示页结构

限制：

- 仍然只作为“附加演示稿”
- 不覆盖原始 html 文件
- 不改现有预览链路默认行为

### B2-2：幕内动画系统

目标：

- 每幕支持元素分步出现
- 支持标题、卡片、图表、注释区的渐进展示
- 支持当前幕内部的 highlight/focus

限制：

- 动画必须可关闭
- 默认动画时长收敛，避免影响语音播报节奏
- 不依赖主进程高频驱动

### B2-3：视觉模板库

目标：

- 提供封面幕、总览幕、步骤幕、对比幕、数据亮点幕、总结幕等模板
- 根据 scene 内容选择合适模板

限制：

- 先做少量高置信模板
- 不做开放式自由布局生成

### B2-4：问答跳幕

目标：

- 用户提问题后，跳到最相关的一幕
- 支持“重讲这一幕”“跳到总结”“回到第一页”等操作

限制：

- 问题到幕的映射先走保守匹配
- 不影响现有普通对话指令解析

当前落地状态：

- 已完成保守版 scene 匹配
- 当前只支持显式页码/总结/当前幕，以及高置信标题关键词命中
- 仍未做基于语义理解的自由问答跳幕

### B2-5：讲稿增强

目标：

- 让每幕 narration 更自然
- 支持“简洁播报 / 详细讲解”两档

限制：

- 仍与现有“自动朗读助手回复”互斥
- 不改现有全局 TTS 开关语义

当前落地状态：

- 已支持 `摘要播报 / 详细讲解`
- 详细模式目前仍基于现有 scene `title + summary` 做保守扩写
- 尚未引入更复杂的讲稿重写或风格模板

## 进入 B2 前的前置门槛

只有满足以下条件，才建议继续推进 B2：

- B1 的演示稿生成链路稳定
- “演示上面的 html”可以稳定进入逐幕演示
- 暂停/继续/停止链路稳定
- 与现有 TTS 自动播放不存在冲突
- `masterEnabled=false` 时新增能力对现有语音链路零影响
