# QingShuClaw 青数模块化接入验收清单

## 1. 文档目标

本文用于提供一份可直接执行的手工验收清单，覆盖当前这轮“青数模块化接入与治理入口”落地后的主要检查项。

重点关注三类目标：

- 正式链路不回归
- 开发态治理入口边界正确
- 模块关闭后的降级行为正确

配套说明文档：

- `docs/qingshu-modular-governance-overview.md`

## 2. 验收前准备

建议先确认以下前置条件：

- 当前分支为目标联调分支，例如 `qingshu-dev-alive`
- 本地可正常启动 `QingShuClaw`
- 若需验证开发态入口，使用开发环境启动
- 若需验证青数模块降级，准备可切换的 `qingshuModules` 配置

建议至少执行一次静态检查：

- `npm run compile:electron`
- `npx tsc --noEmit -p tsconfig.json`

## 3. 正式链路不回归

### 3.1 登录与主链路

检查项：

1. 启动客户端，确认应用可正常进入主界面。
2. 使用已有登录方式验证登录主链路未受影响。
3. 登录成功后确认用户信息、额度、模型等基础数据仍可正常加载。

预期结果：

- 未因治理入口新增而影响登录
- 未出现启动异常、白屏或主流程阻断

### 3.2 Skill 管理主链路

检查项：

1. 打开技能管理页。
2. 验证已安装技能列表可正常显示。
3. 对普通 skill 执行启用 / 禁用。
4. 对非内置 skill 执行删除流程。
5. 使用原有方式执行 ZIP / 本地目录 / GitHub 导入。

预期结果：

- 原有 skill 管理动作不受影响
- 新增治理能力不替代原有导入或删除逻辑

### 3.3 Agent 管理主链路

检查项：

1. 打开 Agent 创建弹窗。
2. 正常填写基础信息并创建 Agent。
3. 打开已有 Agent 设置面板。
4. 修改名称、描述、技能并保存。

预期结果：

- 创建与更新流程正常
- 保存行为不依赖开发态治理入口

## 4. 开发态治理入口

以下检查仅在开发环境下执行。

### 4.1 Skill 详情治理预览

检查项：

1. 打开技能管理页。
2. 点击某个已安装 skill 进入详情弹窗。
3. 确认出现“治理预览”入口。
4. 打开治理预览，检查依赖声明、校验结果、contract 是否正常展示。
5. 关闭 skill 详情弹窗，确认治理预览一并关闭或保持预期状态。

预期结果：

- 仅开发环境可见
- 只读展示
- 不影响 skill 启用、禁用、删除

### 4.2 Skill 导入前本地治理分析

检查项：

1. 打开“新增技能”菜单。
2. 确认出现：
   - 分析本地 `SKILL.md`
   - 分析本地技能目录
3. 选择一个本地 `SKILL.md` 文件进行分析。
4. 选择一个本地技能目录进行分析。
5. 检查结果弹窗是否正确展示治理结果。
6. 再使用原有导入按钮执行一次正常导入。

预期结果：

- 分析入口仅开发环境可见
- 分析不触发安装
- 原有导入链路保持不变

## 5. Agent Skills 页签调试区

以下检查仅在开发环境下执行。

### 5.1 说明与信息分层

检查项：

1. 打开 Agent 创建页或 Agent 设置页的 Skills 页签。
2. 确认能看到开发态说明块。
3. 确认页面语义清晰区分：
   - 已保存 bundles
   - 本地草稿 bundles
   - 治理摘要联动区

预期结果：

- 不需要依赖猜测即可理解三层关系

### 5.2 已保存与草稿分离

检查项：

1. 打开一个已有 `toolBundleIds` 的 Agent。
2. 记录只读区展示的已保存 bundles。
3. 在调试选择器中增删若干 bundle。
4. 观察状态提示是否变化为：
   - `草稿未保存`
   - 已保存数量与草稿数量发生变化

预期结果：

- 只读区仍展示真实保存值
- 调试选择器只影响草稿
- 草稿状态提示明确可见

### 5.3 治理摘要联动

检查项：

1. 在 Skills 页签中选择一组声明了 bundle 依赖的 skills。
2. 观察治理摘要中的：
   - 依赖 bundles
   - 当前 bundles
   - 缺失 bundles
3. 在调试选择器中逐步补齐缺失 bundles。
4. 观察治理摘要是否实时变化。

预期结果：

- 治理摘要随草稿实时联动
- 不需要保存即可观察差异

### 5.4 不写回正式保存

检查项：

1. 在调试选择器中修改草稿 bundles。
2. 不修改正式持久化逻辑，仅保存 Agent 其它字段或直接关闭页面。
3. 重新打开该 Agent 的 Skills 页签。

预期结果：

- 只读区仍显示原始已保存 bundles
- 草稿不会自动写回 `toolBundleIds`

## 6. 模块关闭降级

### 6.1 关闭 module

检查项：

1. 关闭某个 QingShu module。
2. 重启应用或重新加载对应配置。
3. 打开开发态 bundle 调试区。
4. 检查该 module 对应 bundles 是否从可选列表中消失。

预期结果：

- 对应 shared tools 与 bundles 不再暴露
- 其它模块能力不受影响

### 6.2 sharedToolsEnabled = false

检查项：

1. 保持 module 为启用状态。
2. 将 `sharedToolsEnabled` 设为 `false`。
3. 重新进入相关治理调试入口。

预期结果：

- module 仍可处于启用状态
- 但 bundles / shared tools 不可见
- 治理层不应继续将该 module 视为可用工具来源

### 6.3 catalog 为空时 fail closed

检查项：

1. 让当前 catalog 不包含目标 bundles / tools。
2. 打开一个声明了对应依赖的 skill 治理预览。
3. 查看治理校验结果。

预期结果：

- 返回缺失类问题
- 不会错误显示“可用”

## 7. OpenClaw 与旧链路回归

检查项：

1. 启动 OpenClaw 主流程。
2. 验证原有会话、技能、运行链路仍可使用。
3. 验证未显式启用 QingShu module 时，不会向普通运行路径额外暴露青数 bundles。

预期结果：

- OpenClaw 主链路不回归
- 旧 skill / plugin / agent 主链路不被治理入口干扰

## 8. 建议验收记录格式

建议按如下格式记录：

- 验收日期
- 验收分支
- 环境类型：开发 / 测试
- 模块开关状态
- 通过项
- 未通过项
- 备注与截图路径

## 9. 当前阶段的验收结论口径

若以上检查项全部通过，可认为当前阶段已满足：

- 正式链路稳定
- 开发态治理入口边界清晰
- 模块关闭可安全降级
- 后续可继续在此基础上推进正式 bundle 编辑能力，而不必回头重构当前治理基础层
