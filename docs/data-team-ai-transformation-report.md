# 数据团队 AI 化规划汇报材料（面向 CEO / CTO）

## 文档说明

本文档分为两部分：

1. `反驳者视角`：以资深 CTO、且熟悉 AI 落地的视角，对当前规划提出最关键的挑战
2. `正式汇报稿`：将这些挑战吸收后，整理出一版适合向 CEO / CTO 进行正式汇报的材料

这样处理的目的，不是否定规划，而是帮助这套规划从“方向正确”进一步走到“可以落地、可以评估、可以拿资源”。

---

## 第一部分：反驳者视角的核心挑战

如果我以一个强势、谨慎、结果导向的 CTO 身份来听这份汇报，我会重点挑战以下问题。

### 1. 你们其实在同时做三场战争，而不是一场战争

你们当前规划同时覆盖了三类事情：

- 底层数仓与元数据基建
- 类 OpenClaw 的自研运行时与 Skill Hub
- 数据分析师全员 Skill 化的组织运动

这三件事每一件都足以成为单独年度主题。把它们一起推进，风险不是技术难，而是节奏失控、责任失焦、验收模糊。

我会追问：

- 三件事中，哪一件是“必须先赢”的主矛盾？
- 如果资源只够打一件事，另外两件如何降级？
- 如果 6 个月后只允许保住一个成果，你们要保住什么？

### 2. 你们说的是“技能沉淀”，但我担心最后做成“技能仓库堆积”

很多团队一讲 Skill Hub，就会自然滑向“做一个平台，把各种能力放进去”。

问题在于：

- 有多少 Skill 真正被反复调用？
- 有多少 Skill 能稳定产出高质量结果？
- 有多少 Skill 只是作者本人能用？

我会质疑：

- Skill 的最小有效单元到底是什么？
- 什么叫“一个 Skill 成功”？
- Skill 的淘汰、归并、升级机制是什么？
- 你们如何防止 Skill Hub 变成新的知识坟场？

### 3. 你们为什么要自研类似 OpenClaw 的架构，而不是先复用现成框架？

“为了沉淀和复用分析师技能”是一个成立的动机，但它还不足以直接证明“必须自研框架”。

我会继续追问：

- 现成框架到底满足不了什么？
- 是权限治理不够、运行时不可控、插件生态不匹配，还是数据接入链路不适合？
- 你们自研的边界在哪里？
- 是自研运行时，还是只自研 Skill Hub、Tool 管理和治理层？

如果这个边界讲不清，极容易变成“大而全平台工程”。

### 4. 你们说客户成功链路是必赢场景，但还没有充分证明它是“最先该打”的战场

你们当前优先级判断是合理的，但从 CTO/CEO 视角还需要更硬的论证。

我会问：

- 为什么是客户成功链路，而不是抢班调控自动化或 C 端增长？
- 这个场景的业务收益如何量化？
- 它最适合 AI 化的原因是什么？
- 它的落地阻力是不是最低？
- 它成功后是否能反哺其它场景？

如果没有这层论证，优先级就更像直觉，而不是经营判断。

### 5. 你们说“不统一所有指标，只统一核心场景下的核心实体和事实”，这是正确方向，但还不够具体

这是一个我认可的收敛动作，符合 `KISS` 和 `YAGNI`。

但我会继续追问：

- 哪些实体是第一阶段必须统一的？
- 哪些事实是客户成功链路绕不过去的？
- 跨场景核心语义的边界怎么定义？
- 谁拥有这些定义的最终解释权？

如果这些问题没有具体答案，后面 Skill、Tool、Agent 全都会建立在漂浮语义上。

### 6. 你们强调分析师隐性知识沉淀，但隐性知识不是“写下来”就完成了

这是这份规划里最有价值的部分，也是最容易低估难度的部分。

分析师的隐性知识包括：

- 分析路径判断
- 异常识别经验
- 变量筛选习惯
- 结论表达方式
- 面向不同对象的汇报逻辑

这些东西并不是简单写成一段 Prompt 或一份 SQL 就能复制。

我会挑战：

- 你们如何把“经验”结构化？
- 你们沉淀的是 SQL、模板、分析框架，还是决策规则？
- 优秀分析师写出来的 Skill，如何被别人安全使用？
- 如果 Skill 产出质量差异很大，平台如何做分层推荐？

### 7. 你们说“全员 Skill 化是一场运动”，但组织运动最怕口号化

“全员 Skill 化”可以是方向，但不能是 KPI 口号。

我会反驳：

- 如果所有人都要 Skill 化，会不会把稀缺的关键人也拖进低效试错？
- Skill 化的目标是行为变化，还是结果变化？
- 是要求每个人都写 Skill，还是要求每个人都能用 Skill、改 Skill、发现可 Skill 化机会？

如果这个定义不清晰，最后会出现：

- 有人为了完成动作去写低质量 Skill
- 有人把原本已经高效的工作方式硬拧成 Skill
- 平台侧获得大量“形式上的活跃”，却没有业务价值

### 8. 评测体系是这套方案成败的生死线，但现在还只是一个“意识到要做”

你们已经意识到需要评测标准，这很好。

但从 CTO 视角，这不是配套项，而是主干项。

我会追问：

- Skill 的评测维度是什么？准确性、稳定性、可解释性、耗时、调用率、转化率？
- 谁来评？平台自动评、人工评、业务评，还是混合？
- 什么样的 Skill 才能进入推荐池？
- 什么样的 Skill 必须被下架？
- 一个 Skill 更新后，如何知道版本有没有退化？

没有评测体系，Skill Hub 不是资产系统，只是分发系统。

### 9. 你们讲的是效率提升，但 CEO 更关心“经营结果提升”

如果你们只讲：

- 分析更快了
- 汇报更标准了
- 客成效率更高了

这还不够。

CEO 更关心：

- 是否提高客户续约率、试点转化率、复购率？
- 是否缩短售前到试点的价值证明周期？
- 是否让中高水平客成的方法被复制到更多人身上？

也就是说，你们必须从“工具效率”翻译到“业务结果”。

### 10. 你们最大的潜在风险，不是做不出来，而是做成“半平台、半项目、半运动”

这是我最担心的结局：

- 平台没做透，只够跑几个样板
- 业务没打透，停留在演示型 Agent
- 组织没转成，只是阶段性热情

所以从反驳者视角，我的结论是：

这套规划方向对，但现在必须极度收敛，先把“客户成功链路分析 Agent 化”打成一个真正的样板胜仗，把它当作主战场，其它两条线全部围绕它服务。

## 第二部分：正式汇报稿（面向 CEO / CTO）

## 1. 汇报主题

数据团队 AI 化规划：以客户成功链路为第一主战场，建设可治理、可复用、可复制的数据分析能力体系。

## 2. 一句话结论

我们的目标不是简单做几个 Agent，也不是搭一个 Skill 平台，而是把数据团队长期沉淀的分析能力，系统化地转化成企业可复用的 AI 能力资产，并优先在客户成功链路上形成可证明业务价值的突破。

## 3. 当前规划的三个重点

当前规划包含三项工作：

1. 底层数仓基建与元数据层建设
2. 基于数仓能力建设统一的 Skill / Tool / Agent 运行与沉淀体系
3. 推动数据分析师团队逐步 Skill 化

其中，这三项并非并列推进，而应围绕第一主战场形成主次关系。

## 4. 第一主战场：客户成功链路分析 Agent 化

我们判断，当前最值得优先投入的场景，是面向客户成功团队的客户全链路分析 Agent 化。

主要覆盖：

- 售前分析
- 售中阶段性报告
- 试点阶段关键价值复盘

选择该场景作为第一主战场，原因有四点：

1. 业务价值明确  
   这一链路直接连接客户成交、试点推进、价值证明与续约可能性。

2. 能力复制价值高  
   当前客户成功经理之间能力参差不齐，高质量 Agent 可以显著拉齐方法与输出标准。

3. 组织协同阻力相对较低  
   相比抢班自动化等跨团队复杂协同场景，该场景更容易形成闭环。

4. 对后续场景有示范作用  
   如果这一场景成功，后续可复制到更多业务链路，而不是从零开始。

## 5. 战略判断：先统一“关键场景语义”，不追求一次性统一全域指标

我们不主张第一阶段统一所有 B 端、C 端、客户侧、用户侧的全部关键指标。

第一阶段的策略是：

- 基于客户成功核心场景
- 统一该场景下最关键的基础实体和核心事实
- 对齐跨场景可复用的核心语义

这样做有三个好处：

- 符合 `KISS`：避免一开始进入过大范围治理
- 符合 `YAGNI`：先解决主战场所需，而不是为假想未来做过度建设
- 降低落地风险：确保模型、数据、Skill 和 Agent 都建立在清晰的业务语义上

## 6. 为什么要建设统一的 Skill / Tool / Agent 沉淀体系

我们的核心判断是：

客户成功的全流程分析工作，不可能通过一个“大 Skill”一次性解决。

更现实的方式是：

- 将全流程拆解为多个高频、稳定、可复用的分析环节
- 将这些环节继续拆解成更细颗粒度的 Skill / Tool
- 通过统一的中心进行沉淀、发布、推荐、迭代和治理

因此，我们需要一个统一的 Skill Hub，其价值不只是“放 Skill”，而是：

- 统一发布分析能力
- 形成最佳实践库
- 监控调用情况和效果差异
- 快速推广高质量方法
- 让优秀分析师的隐性知识被更多人复用

## 7. Skill Hub 的真正价值，不是“知识存储”，而是“知识复制”

我们认为 Skill 的价值不局限于低阶问题自动化。

不同分析师写出的 Skill，最终产出质量差异会很大，这恰恰说明：

- Skill 是分析能力的载体
- Skill 可以承接分析师的隐性知识
- Skill 的差异本身，就是组织能力差异的显性化结果

因此，Skill Hub 的核心目标不是堆积 Skill，而是让组织看清：

- 哪些 Skill 真正有价值
- 哪些分析方法值得推广
- 哪些培训真的转化为了实际能力

## 8. 分析师全员 Skill 化的组织定位

“全员 Skill 化”不是要求每个人都成为平台开发者，而是推动整个数据团队进入新的工作方式。

组织策略应为：

- 关键场景由关键人员率先攻坚
- 先进者带动后进者
- 后进者至少先行动起来，进入高频试错状态

这更像是一场组织升级，而不是单纯技术改造。

我们的判断是：

- 传统工作方式目前仍能支撑业务
- 但如果不尽快进入 AI 化工作范式，团队的职业路径和能力上限会受到限制
- 整体转向风险可控，关键在于以主战场胜仗带动组织转变

## 9. 需要同步建设的关键机制

为了避免 Skill Hub 变成堆积平台，必须同步建设以下机制：

### 9.1 评测机制

建立 Skill 质量评测标准，至少覆盖：

- 结果准确性
- 输出稳定性
- 可解释性
- 调用频率
- 业务使用效果

### 9.2 推荐机制

针对客户成功团队，推荐经过验证的优秀 Skill，而不是让一线同学自己在大量 Skill 中试错。

### 9.3 培训闭环机制

培训的价值不再停留在“是否参加过”，而是可以通过 Skill 调用、使用效果和业务落地情况直接验证。

### 9.4 生命周期治理机制

Skill 必须有：

- 发布
- 评测
- 升级
- 淘汰
- 归并

否则平台会快速失控。

## 10. 第一阶段建议的工作重心

第一阶段建议集中做三件事：

1. 围绕客户成功链路，统一最关键的业务实体、核心事实与分析方法
2. 打通一套可运行的 Skill / Tool / Agent 沉淀与发布闭环
3. 做出少量但高质量、可被真实使用的样板 Agent

不建议第一阶段：

- 追求全域指标统一
- 同时全面推进多个复杂业务战场
- 将全员 Skill 化做成运动式 KPI

## 11. 对 CEO / CTO 的资源与决策诉求

如果希望这项规划真正落地，我们建议高层明确支持以下决策：

1. 确认客户成功链路为第一主战场  
   其它方向暂不与之争资源优先级。

2. 确认阶段性成功标准  
   第一阶段以“客户成功链路的高质量 Agent 样板胜利”为核心验收，而非平台功能数。

3. 支持同步建设评测与治理机制  
   Skill Hub 不是单独产品功能，而是组织能力资产平台，必须从第一天就带治理。

4. 支持关键人才集中投入  
   关键样板必须由关键人员攻坚，不能平均分配。

## 12. 最终想达成的结果

短期看，我们要实现的是：

- 客户成功链路上若干高质量 Agent 落地
- 高水平分析能力开始被复制
- 客成团队效率和输出稳定性显著提升

中期看，我们要形成的是：

- 统一的 Skill / Tool / Agent 沉淀中心
- 分析能力的版本化、评测化和推荐机制
- 数据团队从“项目型支持”走向“能力型供给”

长期看，我们要构建的是：

- 企业自己的分析知识库
- 与 Tool 绑定的 SQL / 方法论沉淀
- 能够支撑 AI 自动生成 SQL、自动规划 Skill、自动生成 Tool 的能力底座

这意味着，今天建设的不是一个局部工具，而是未来企业数据 AI 能力体系的地基。
