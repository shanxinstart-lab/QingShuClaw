# QingShuClaw 界面变更说明

## 1. 文档目的

这份文档用于把当前 `QingShuClaw` 工作台界面的改版结果沉淀成一份可跨分支复用的实施说明。目标不是描述业务新功能，而是帮助任意分支在**不改变既有核心逻辑和能力**的前提下，把前端界面与交互恢复到当前版本的样子。

本次改版遵循以下原则：

- `KISS`：优先复用现有页面、IPC、Redux、Agent 能力，不新造平行逻辑。
- `YAGNI`：不新增“任务/技能/应用/对话”的底层业务能力，只改展示与交互。
- `DRY`：优先复用已有登录、会话、Agent 设置、会话重命名/删除等能力。
- `SOLID`：将展示层改动收敛在布局组件和少量类型扩展，避免业务耦合扩散。

---

## 2. 总体改版目标

### 2.1 工作台骨架

将原始界面重构为统一的工作台三栏布局：

1. 一级侧边栏：主导航、用户区、设置、品牌标识。
2. 二级侧边栏：对话模块的搜索、创建 Agent、Agent 分组、最近会话。
3. 主内容区：根据主导航切换为对话、任务、技能、应用四个视图。

### 2.2 改版边界

保留原有能力：

- 登录/登出、用户信息、额度展示
- 对话创建、继续对话、删除、重命名、置顶
- Agent 管理、技能管理、应用连接、任务列表
- IM 通道会话映射与历史归档

不新增能力：

- 不新增新的业务实体
- 不新增新的后端接口
- 不改变原有任务、技能、应用、会话的业务规则

如果要在任意分支复刻当前界面，建议按这个顺序实施：

1. 先接入三栏工作台骨架
   - `App.tsx`
   - `PrimarySidebar.tsx`

3. 接入任务、技能、应用三个主内容区
4. 恢复登录区与顶部用户态
5. 补 Agent 组与会话列表细节
   - `青数CLAW` logo
   - 内置 Agent 标签
   - 会话来源图标
   - 会话三点菜单
6. 最后补齐会话刷新广播与会话来源元数据

这样做的原因：

- 先搭骨架最符合 `KISS`
- 会话来源与刷新属于“细节接线”，放在最后风险更低
- 避免先改会话列表，后面骨架重构时再次返工

## 8. 预设 Agent 在新建 Agent 中的利用方式

当前实现状态：

- 预设 Agent 展示与直接安装逻辑仍保留在 [src/renderer/components/agent/AgentsView.tsx](/Users/wuyongsheng/workspace/projects/QingShuClaw/src/renderer/components/agent/AgentsView.tsx)
- 新建 Agent 弹窗 [src/renderer/components/agent/AgentCreateModal.tsx](/Users/wuyongsheng/workspace/projects/QingShuClaw/src/renderer/components/agent/AgentCreateModal.tsx) 已新增“预设模板创建”入口

也就是说，现在有两条路径：

1. 在 Agent 列表页直接安装一个预设 Agent
2. 在“新建 Agent”弹窗里选择一个预设模板，自动预填后继续编辑，最后保存成自己的自定义 Agent

### 8.1 当前弹窗交互

`AgentCreateModal` 的“基础信息”页顶部新增了“创建方式”区域：

1. 从空白创建
2. 从预设模板创建

当用户切换到“从预设模板创建”时：

- 弹窗顶部会展示所有预设 Agent 模板
- 已安装的预设会显示“已安装”标签，但仍然允许作为模板使用
- 点击某个模板后，自动预填以下字段：
  - `name`
  - `description`
  - `systemPrompt`
  - `icon`
  - `skillIds`
- 同时重置当前草稿里的：
  - `identity`
  - `toolBundleIds`
  - `debugToolBundleIds`
  - 治理校验的 warning 状态

设计意图：

- “新建”保持创建心智，不等于“直接安装”
- 预设被当作模板，而不是独立的只读成品
- 用户可以在模板基础上继续修改并沉淀成自己的 Agent

### 8.2 当前保留的老入口

为了兼顾已有用户习惯，当前没有移除 `AgentsView` 中的预设 Agent 入口：

- 在 Agent 列表页，用户仍可直接点“添加”安装预设 Agent
- 这条路径适合想快速启用官方预设、不需要修改内容的场景

换句话说：

- `AgentsView` 负责“直接启用预设”
- `AgentCreateModal` 负责“基于预设模板创建自定义 Agent”

### 8.3 为什么这样做

这是当前更合适的方案，原因如下：

- `KISS`：复用 `agentService.getPresets()`，不新增后端接口
- `YAGNI`：不新增新的 Agent 类型，只做模板预填
- `DRY`：不重复维护第二套创建表单，只在现有表单顶部补模板选择
- 用户心智更顺：新建就是“创建自己的 Agent”，预设只是更快的起点

### 8.4 后续可选优化

如果继续优化这块，建议按以下顺序推进：

1. 在模板卡片上展示预设包含的技能数量或技能名称摘要
2. 支持“最近使用模板”排序
3. 允许在模板选中后显示“与空白创建的差异提示”
4. 若未来要收敛入口，再评估是否弱化 `AgentsView` 里的“直接安装预设”区域

### 11.12 登录态恢复与套餐模型注入时机调整

本轮补充调整：

- 检查后确认，登录后 `lobster-ai` 套餐模型进入模型选项的唯一入口仍然是 `authService.init()` 内部的 `loadServerModels()`
- `ModelSelector` 本身并没有过滤服务端模型；只要模型成功注入 `model.availableModels`，就会在下拉中以“套餐模型”分组显示
- 为了此前首屏提速，工作台曾把 `authService.init()` 延后到壳层 ready 之后才开始，导致登录态恢复与套餐模型注入整体后移

- 这样可以兼顾两点：
  - 工作台仍然尽快出现
  - 登录用户的套餐模型更早回到模型选项中

验收标准：

- 已登录用户打开工作台后，模型选择器应更早出现 `lobster-ai` / 套餐模型相关选项
- 首屏打开速度不应退回到旧的长时间“启动中”状态
