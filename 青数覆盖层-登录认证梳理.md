# 青数覆盖层：登录认证梳理

## 1. 背景与目标

这份文档用于把当前分支里“相对 `main` 需要刻意保留”的青数登录认证链路、登录承接 UI、跨端桥接和登录态联动一起抽出来，作为后续从最新 `main` 新拉分支后的回贴清单。

这里关注的是：

1. 登录认证主干
2. 主进程认证适配层与配置
3. renderer 认证服务与状态管理
4. 登录入口、欢迎浮层、设置页认证配置等 UI 承接层
5. Web / Desktop bridge、deep link 与 Portal 跳转
6. 登录态与 `server models`、`qingshuManaged` 的联动

这份文档不重复解释整套平台认证协议细节。协议、接口与会话口径以：

1. `docs/qingshu-auth-bridge-overview.md`
2. `docs/青数飞书登录认证.md`

为事实来源。

## 2. 核心结论

当前分支的登录认证覆盖层不是“一个登录按钮”，而是一组必须整体保留的业务承接链：

1. `src/common/auth.ts` 统一了 renderer / main 共用的认证契约。
2. `src/main/auth/* + src/main/main.ts` 负责桌面端认证适配、token 落地、飞书扫码、bridge 票据、deep link 承接和认证窗口。
3. `src/renderer/services/auth.ts + authSlice.ts` 负责登录态恢复、背景 hydration、退出后本地清理、模型与治理同步。
4. `LoginButton + LoginWelcomeOverlay + Settings` 负责登录前后承接体验，不是可随意拆掉的点状 UI。
5. 顶层 `App.tsx` 同时承接登录欢迎浮层与唤醒浮层，它们都属于当前工作台壳层的一部分。

后续从最新 `main` 新起分支时，这一域必须整组回贴，不能只迁：

1. `LoginButton.tsx`
2. `auth.ts`
3. 某几个 IPC handler

否则容易出现“能打开登录入口，但登录后状态、模型、治理、欢迎浮层或跨端免登断链”的半成品状态。

这套设计也符合当前阶段的工程原则：

1. `KISS`：复用平台用户与平台权限，不再额外造一套桌面账号体系。
2. `YAGNI`：不强行统一所有 token 内部格式，只统一认证语义、桥接能力与落地入口。
3. `SOLID`：把契约、主进程适配层、renderer 服务、UI 承接层拆开，各司其职。
4. `DRY`：共享契约和认证文档作为单一事实来源，避免 Web / Desktop 两侧重复定义口径。

## 3. 认证契约与配置口径

### 3.1 Shared contract

核心文件：

1. `src/common/auth.ts`

当前定义了：

1. `AuthBackend`
2. `AuthConfig`
3. `AuthLoginMode`
4. `BridgeTarget`
5. `CreateBridgeTicketRequest / Response`
6. `ExchangeBridgeCodeRequest`
7. `FeishuScanSessionStatus`
8. `FeishuScanSession / FeishuScanSessionPollResult`
9. `DEFAULT_QTB_API_BASE_URL`
10. `DEFAULT_QTB_WEB_BASE_URL`

这部分是整个认证覆盖层的共享契约，后续回贴时必须保留为单一来源。

### 3.2 主进程配置解析

核心文件：

1. `src/main/auth/config.ts`

当前职责：

1. 从 `app_config.auth` 读取配置
2. 标准化 `qtbApiBaseUrl`
3. 标准化 `qtbWebBaseUrl`
4. 决定当前 `backend`
5. 兼容历史 `legacy_lobster` 口径

### 3.3 设置页落点

核心文件：

1. `src/renderer/components/Settings.tsx`

当前提供：

1. `QTB API Base URL`
2. `QTB Web Base URL`

这意味着登录认证不只是运行时能力，也已经有明确的工作台配置入口。

## 4. 主进程认证适配层

### 4.1 认证适配器

核心文件：

1. `src/main/auth/adapter.ts`

当前已承接：

1. 账号密码登录
2. 飞书扫码会话创建
3. 飞书扫码轮询
4. 飞书授权 URL 获取与外部打开
5. deep link `code` 兑换
6. `refreshToken`
7. `logout`
8. 当前用户获取
9. 套餐 / 额度获取
10. 头像摘要获取
11. 服务端模型获取
12. bridge 票据创建与 bridge code 兑换
13. 本地 token 落地、失效清理与刷新

它是桌面端登录认证的真实业务主干，不应在后续迁移时被散落到多个不一致实现中。

### 4.2 主进程承接窗口与 IPC

核心文件：

1. `src/main/main.ts`
2. `src/main/preload.ts`

当前承接了：

1. 飞书扫码承载窗口 `authWindow`
2. deep link `lobsterai://auth/callback`
3. deep link `lobsterai://auth/bridge`
4. `auth:getPendingCallback`
5. `auth:getPendingBridgeCode`
6. `auth:login`
7. `auth:loginWithPassword`
8. `auth:openFeishuScanWindow`
9. `auth:createFeishuScanSession`
10. `auth:pollFeishuScanSession`
11. `auth:exchange`
12. `auth:createBridgeTicket`
13. `auth:exchangeBridgeCode`
14. `auth:getUser`
15. `auth:getQuota`
16. `auth:getProfileSummary`
17. `auth:logout`
18. `auth:refreshToken`
19. `auth:getAccessToken`
20. `auth:getModels`

这部分决定了 renderer 是否真能访问完整认证链，不能只迁 service 而漏掉 `preload` 和 `main.ts`。

## 5. Renderer 认证服务与状态管理

### 5.1 认证服务

核心文件：

1. `src/renderer/services/auth.ts`

当前职责：

1. 应用启动时恢复登录态
2. 订阅 deep link callback 与 bridge code
3. 创建并缓存飞书扫码会话
4. 轮询扫码状态
5. 账号密码登录
6. 浏览器 / 应用内登录入口承接
7. 创建 bridge ticket
8. 处理 bridge code 兑换
9. 获取 profile summary
10. 获取额度
11. 获取可用服务端模型
12. `logout`
13. 认证失效后清理本地状态
14. 背景 hydration：同步 `server models` 与 `qingshuManaged`

### 5.2 状态切片

核心文件：

1. `src/renderer/store/slices/authSlice.ts`

当前承接：

1. `isLoggedIn`
2. `user`
3. `quota`
4. `profileSummary`
5. loading 状态

它和 `auth.ts`、`LoginButton.tsx` 是一体的；后续不能只迁 UI 不迁状态切片。

### 5.3 退出与清理联动

核心文件：

1. `src/renderer/services/auth.ts`
2. `src/renderer/services/authSessionReset.ts`

当前会在退出或 session 失效时：

1. 禁用 `QingShu Managed` 相关可见项
2. 清空当前会话
3. 清空 server models
4. 重置当前 agent
5. 回退到未登录态

这层是“青数内置治理链”和“登录认证”之间的连接点，必须保留。

## 6. 登录 UI 承接层

### 6.1 登录入口与身份区

核心文件：

1. `src/renderer/components/LoginButton.tsx`
2. `src/renderer/components/layout/PrimarySidebar.tsx`

当前承接：

1. 账号密码登录
2. 飞书扫码登录
3. 应用内扫码页入口
4. 浏览器登录入口
5. 已登录后的头像、身份、额度摘要
6. 打开青数 Portal
7. 查看套餐 / 额度
8. 退出登录

这不是孤立按钮，而是当前一级侧栏顶部身份区的一部分。

### 6.2 登录欢迎浮层

核心文件：

1. `src/renderer/components/LoginWelcomeOverlay.tsx`
2. `src/renderer/App.tsx`

当前行为：

1. 登录成功后由 `LoginButton` 触发 `showLoginWelcome`
2. 顶层显示短时欢迎浮层
3. 浮层自动收起
4. 带有明显的青数品牌化动效与视觉承接

这层应被视为登录后的“壳层反馈”，不是可随意删掉的装饰动画。

### 6.3 设置页认证分区

核心文件：

1. `src/renderer/components/Settings.tsx`

当前作用：

1. 暴露认证后端基地址
2. 支撑不同环境下的联调、桥接和扫码配置
3. 作为登录认证覆盖层的运维入口

## 7. 顶层叠层与唤醒浮层关系

核心文件：

1. `src/renderer/App.tsx`
2. `src/renderer/components/LoginWelcomeOverlay.tsx`
3. `src/renderer/components/WakeActivationOverlay.tsx`

当前顶层工作台同时承接：

1. `LoginWelcomeOverlay`
2. `WakeActivationOverlay`
3. `Toast`
4. `Settings`
5. `HistoryDrawer`
6. `SearchModal`

这意味着登录欢迎浮层与唤醒浮层都不是“页面内局部组件”，而是工作台顶层叠层的一部分。后续如果从 `main` 新分支回贴时只迁其中一个，很容易打乱顶层层级、焦点与视觉反馈顺序。

## 8. Web / Desktop Bridge 与 Deep Link

### 8.1 Bridge

核心文件：

1. `src/renderer/services/auth.ts`
2. `src/main/auth/adapter.ts`
3. `src/common/auth.ts`
4. `src/main/main.ts`

当前能力：

1. `createBridgeTicket`
2. `handleBridgeCode`
3. `BridgeTarget.Web`
4. `BridgeTarget.Desktop`

这部分是青数 Web 与桌面端双向免登的关键，不应在主干回贴时丢掉。

### 8.2 Deep link

核心文件：

1. `src/main/main.ts`

当前承接：

1. `lobsterai://auth/callback`
2. `lobsterai://auth/bridge`
3. renderer 初始化后的 pending callback / pending bridge code 回放

即使未来品牌继续演进，这层也不能在未完成兼容替换前被顺手改掉。

## 9. 与模型、治理和工作台的联动

### 9.1 服务端模型联动

核心文件：

1. `src/renderer/services/auth.ts`
2. `src/renderer/store/slices/modelSlice.ts`

当前行为：

1. 登录成功后拉取 server models
2. 退出登录后清空 server models
3. 服务端模型是当前分支 provider / model 入口的一部分

### 9.2 青数治理联动

核心文件：

1. `src/renderer/services/auth.ts`
2. `src/renderer/services/qingshuManaged.ts`

当前行为：

1. 登录成功后背景同步 managed catalog
2. 登录失效后禁用 managed 项
3. 这保证“有权限时可见、失效后收回”的治理语义成立

### 9.3 工作台身份承接

核心文件：

1. `src/renderer/components/LoginButton.tsx`
2. `src/renderer/components/layout/PrimarySidebar.tsx`

当前行为：

1. 身份态直接影响侧栏顶部展示
2. 额度、头像、昵称、入口菜单都依赖登录态
3. 这是当前工作台壳层的重要组成部分

参考文档：

## 11. 验收清单

后续从最新 `main` 新分支回贴完成后，至少检查：

1. 启动后能自动恢复已保存的登录态。
2. 账号密码登录链路可用。
3. 飞书扫码会话创建、二维码展示、轮询登录可用。
4. 登录成功后会显示欢迎浮层，并在预期时间内自动收起。
5. 打开青数 Portal 与 bridge 免登链路可用。
6. 设置页里的 `qtbApiBaseUrl / qtbWebBaseUrl` 修改后生效。
7. 登录成功后 server models 会刷新，退出后会清空。
8. 登录成功后 `qingshuManaged` 会同步，失效后会收回相关可见项。
9. 顶层叠层中登录欢迎浮层与唤醒浮层的显示顺序、层级和焦点表现正常。
