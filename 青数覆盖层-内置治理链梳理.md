# 青数覆盖层：内置治理链梳理

## 1. 背景与目标

这份文档用于把当前分支里“青数内置治理链”单独抽出来，供后续从最新 `main` 新拉分支后回贴使用。

这里说的“内置治理链”，不是单指一个组件，而是以下几层组合：

1. 主进程 `qingshuModules` 模块宿主与治理能力
2. 主进程 managed catalog 与托管对象同步
3. renderer 侧 managed 访问控制与锁定语义
4. renderer 侧 skill / agent 治理预览、bundle 兼容提示与开发态调试入口

当前文档目标不是重复记录研发过程，而是回答两个问题：

1. 哪些能力属于必须保留的治理覆盖层
2. 后续从 `main` 新分支回贴时，哪些文件必须成组迁移

## 2. 核心结论

当前分支的治理链已经不是临时 UI，而是有明确主干边界的一层正式覆盖能力：

1. 主进程已经具备 `qingshuModules` 宿主、治理分析、catalog 摘要与只读 IPC。
2. 托管对象已经不是普通本地 agent / skill，而是有 `managed` 来源、权限态和只读边界。
3. renderer 侧对 managed 对象的“登录后可用 / 暂无权限 / 只读展示 / 本地追加技能”已有稳定语义。
4. 当前分支故意保留了“开发态治理入口”和“正式链路”并存的结构，不能简单拿 `main` 覆盖。

因此，后续回贴时不能只迁其中一层：

1. 只迁主进程治理，不迁 renderer，界面看不到治理语义。
2. 只迁 renderer，不迁主进程，界面能打开但拿不到真实治理结果。

## 3. 主进程治理主干

### 3.1 `qingshuModules` 宿主层

核心文件：

1. `src/main/qingshuModules/host.ts`
2. `src/main/qingshuModules/config.ts`
3. `src/main/qingshuModules/index.ts`

职责：

1. 读取 module feature flag
2. 注册 module
3. 汇总共享 tools
4. 汇总 bundles
5. 生成 shared tool catalog
6. 模块关闭时安全降级

这是当前治理覆盖层的基础设施起点。

### 3.2 治理分析与 contract 主线

核心文件：

1. `src/main/qingshuModules/sharedToolCatalog.ts`
2. `src/main/qingshuModules/skillDependencies.ts`
3. `src/main/qingshuModules/skillDependencyValidator.ts`
4. `src/main/qingshuModules/contractGenerator.ts`
5. `src/main/qingshuModules/skillGovernance.ts`
6. `src/main/qingshuModules/governanceService.ts`

职责：

1. 汇总共享工具目录
2. 解析 skill 依赖声明
3. 校验 `toolBundles / toolRefs / capabilityRefs`
4. 生成 markdown / json contract
5. 输出统一 governance result

### 3.3 主进程入口与 IPC

核心文件：

1. `src/main/main.ts`

当前正式只读 IPC 包括：

1. `skills:governance:analyzeById`
2. `skills:governance:analyzeFiles`
3. `skills:governance:getCatalogSummary`

这部分属于正式链路，不是只给开发态看的临时接口。

## 4. Managed catalog 与托管对象链

### 4.1 managed catalog 服务

核心文件：

1. `src/main/qingshuManaged/catalogService.ts`

职责：

1. 拉取 managed agent / skill / tool 目录
2. 持久化 catalog snapshot
3. 同步允许访问的 managed skill 包
4. 把后端 managed agent 映射成只读本地 `Agent`
5. 维护 managed agent 的本地追加技能

### 4.2 托管对象进入本地数据层

核心文件：

1. `src/main/coworkStore.ts`
2. `src/renderer/services/agent.ts`
3. `src/renderer/services/skill.ts`

当前已经扩展出的关键字段包括：

1. `source = managed`
2. `sourceType = qingshu-managed`
3. `allowed`
4. `policyNote`
5. `managedToolNames`
6. `managedBaseSkillIds`
7. `managedExtraSkillIds`
8. `toolBundleIds`

这意味着 managed 对象已经是当前分支的数据模型一部分，而不是界面层的“假标签”。

## 5. Renderer 侧治理与只读语义

### 5.1 managed 访问控制统一语义

核心文件：

1. `src/renderer/services/qingshuManagedUi.ts`
2. `src/shared/qingshuManaged/access.ts`
3. `src/shared/qingshuManaged/constants.ts`

当前已统一的语义包括：

1. 登录后可用
2. 暂无权限
3. 只读 managed
4. 来源标签映射

后续若继续从 `main` 拉齐，这层 helper 应继续作为唯一语义入口保留。

### 5.2 Skills 页的治理入口

核心文件：

1. `src/renderer/components/skills/SkillsManager.tsx`
2. `src/renderer/components/skills/QingShuGovernancePreview.tsx`
3. `src/renderer/services/qingshuGovernance.ts`

当前保留的行为：

1. managed skill 分可用 / 锁定过滤
2. 治理预览入口
3. skill 导入前治理分析
4. 锁定 tag 与 hint

### 5.3 Agent 页的治理入口

核心文件：

1. `src/renderer/components/agent/AgentSettingsPanel.tsx`
2. `src/renderer/components/agent/AgentCreateModal.tsx`
3. `src/renderer/components/agent/AgentSkillSelector.tsx`
4. `src/renderer/components/agent/AgentSkillGovernancePreview.tsx`
5. `src/renderer/components/agent/AgentToolBundleSelector.tsx`
6. `src/renderer/components/agent/AgentToolBundleCompatibilityHint.tsx`
7. `src/renderer/components/agent/AgentToolBundleReadOnlyPanel.tsx`
8. `src/renderer/services/qingshuGovernanceSummary.ts`

当前保留的行为：

1. managed agent 的只读基线技能展示
2. 本地追加技能入口
3. managed tool 只读展示
4. `toolBundleIds` 的治理兼容提示
5. 开发态 bundle 调试区与治理摘要联动

## 6. 当前边界与故意保留差异

### 6.1 正式链路

当前已经属于正式链路的内容：

1. `qingshuModules` 宿主
2. catalog summary
3. governance service
4. 只读治理 IPC
5. managed catalog 同步
6. managed agent / skill / tool 的访问控制与只读语义

### 6.2 开发态入口

当前故意保留为开发态的内容：

1. skill 详情治理预览
2. skill 导入前治理分析
3. agent bundle 调试选择器
4. 部分治理说明块

这部分不应机械视为“脏代码”，因为它们承担了调试和治理验证角色。

### 6.3 降级边界

治理链当前的关键边界是：

1. 默认不强行接管旧主链路
2. 模块可关闭
3. 模块关闭后能安全降级
4. 单模块失败不阻断应用主流程

这部分是后续回贴时必须保持的架构原则。

## 8. 从 main 新分支回贴时的建议

### 8.1 必须成组迁移的文件

高优先级：

1. `src/main/qingshuModules/*`
2. `src/main/qingshuManaged/catalogService.ts`
3. `src/main/main.ts` 中的治理与 managed IPC
4. `src/main/coworkStore.ts`
5. `src/shared/qingshuManaged/*`
6. `src/renderer/services/qingshuManagedUi.ts`
7. `src/renderer/services/qingshuGovernance.ts`
8. `src/renderer/services/qingshuGovernanceSummary.ts`
9. `src/renderer/services/qingshuToolBundles.ts`
10. `src/renderer/components/skills/QingShuGovernancePreview.tsx`
11. `src/renderer/components/skills/SkillsManager.tsx`
12. `src/renderer/components/agent/AgentSettingsPanel.tsx`
13. `src/renderer/components/agent/AgentCreateModal.tsx`
14. `src/renderer/components/agent/AgentSkillSelector.tsx`
15. `src/renderer/components/agent/AgentToolBundle*.tsx`

### 8.2 推荐顺序

1. 先迁主进程宿主与 managed catalog。
2. 再迁 shared 常量与访问控制 helper。
3. 再迁技能页和 Agent 页的治理 UI。
4. 最后处理开发态 bundle 调试入口是否保留。

## 9. 验收清单

回贴完成后至少检查：

1. managed agent / skill 未登录时显示“登录后可用”，而不是像普通对象一样可直接编辑。
2. 无权限时显示“暂无权限”与 `policyNote`。
3. managed agent 能展示基线技能、内置工具、本地追加技能。
4. skill 治理预览能拉到依赖、issues、contract。
5. agent 的 `toolBundleIds` 与治理摘要、兼容提示联动正常。
6. 关闭某个 QingShu module 后，应用仍能正常运行，相关治理结果与共享工具不会错误暴露。
