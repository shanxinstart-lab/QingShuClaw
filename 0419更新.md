# 0419 更新记录

## 本次更新目标

本轮工作的目标不是简单拉齐代码，而是在保留当前分支业务改动的前提下，把 `OpenClaw` 相关的公共层、插件层和运行时行为尽量向 `main` 分支对齐，并把之前已经暴露出来的几个阻断问题真正收敛掉：

1. 修复 `OpenClaw gateway client` 在新版 runtime 下无法正确加载的问题。
2. 修复 `openclaw.json` 与新版 `OpenClaw` 严格配置校验不兼容的问题。
3. 将第三方插件的安装、复制、加载路径从旧的 `extensions/` 方案切换到更接近 `main` 的 `third-party-extensions/` 方案。
4. 补齐 `Feishu / WeCom / Weixin / Netease Bee / NIM` 等公共插件链路。
5. 清理历史残留插件目录，避免旧插件覆盖或污染新插件加载结果。
6. 建立后续继续跟进 `main` 时更稳定的接入基础。

## 2. 修复 `OpenClaw gateway client` 入口探测

文件：

1. `src/main/libs/openclawEngineManager.ts`
2. `src/main/libs/agentEngine/openclawRuntimeAdapter.ts`

问题背景：

1. 升级到新版 runtime 后，旧逻辑会错误命中 `client-*.js` 或共享 chunk。
2. 命中的模块导出不是合法 `GatewayClient`，导致报错：
   - `Invalid OpenClaw gateway client module: ... (exports: n, r, t)`

本次处理：

1. 优先尝试新版稳定入口 `dist/plugin-sdk/gateway-runtime.js`。
2. 保留 `method-scopes-*.js` 的兼容探测。
3. 用导出形态校验确认模块里是否真的暴露了 `GatewayClient`。
4. 避免继续把任意 chunk 当成客户端入口。

结果：

1. 不再出现 `Invalid OpenClaw gateway client module`。
2. 实际启动日志已出现：
   - `resolved gateway client entry`
   - `GatewayClient: onHelloOk — handshake succeeded`

## 4. 第三方插件目录切换为 `third-party-extensions`

文件：

1. `scripts/ensure-openclaw-plugins.cjs`
2. `scripts/sync-local-openclaw-extensions.cjs`
3. `src/main/libs/openclawLocalExtensions.ts`
4. `src/main/libs/openclawEngineManager.ts`

问题背景：

1. 历史版本会把第三方插件放到 `extensions/`。
2. 新版网关在 bundle 模式下仍可能扫描 `extensions/`，把这些第三方插件误当作 bundled channel。
3. 结果是：
   - 插件找到了，但加载合同不匹配
   - 启动时出现误加载、重复加载、耗时增加或报错

本次处理：

1. 安装脚本改为将第三方插件统一复制到：
   - `vendor/openclaw-runtime/current/third-party-extensions/`
2. 本地自定义扩展同步脚本也切到同一目录。
3. 运行时的插件发现逻辑改为优先从 `third-party-extensions` 读取。
4. 通过 manifest 里的 `id` 识别真实插件身份，不再只看目录名。

结果：

1. 第三方插件来源和加载路径与 `main` 的方向一致。
2. 后续继续升级时，公共插件链路更稳定，不容易再被旧目录污染。

## 下次 `main` 分支再有更新，能否无缝接入？

结论：

不能保证“完全无缝”，但已经具备“基本可顺滑接入”的条件，难度和风险都比这次低很多。

原因分成两部分。

### 已经具备顺滑接入条件的部分

1. 第三方插件目录已经统一到 `third-party-extensions`，目录结构和 `main` 的方向一致。
2. 旧残留插件目录已经清理，减少了“升级后旧插件反向污染”的风险。
3. 插件发现逻辑已经支持真实 manifest id，不再过度依赖目录名。
4. 配置生成逻辑已经只写入“当前已安装且已启用”的插件，减少了升级后因插件缺失导致的硬错误。
5. `ClawHub` 授权链路已经打通，后续如果 token 仍有效，NIM 插件安装会顺畅很多。
6. `GatewayClient` 入口探测兼容逻辑已补齐，新版 runtime 再升级时不太容易重现这次的客户端入口问题。

### 仍然不能承诺绝对无缝的部分

1. `main` 未来如果继续变更插件 id、manifest 结构或 `openclaw.json` schema，仍需要人工复核。
2. `moltbot-popo` 仍受私有源网络影响，这个外部依赖没有被根治。
3. `ClawHub` 社区插件如果再次调整真实 manifest id，当前兼容映射也需要同步更新。
4. 当前分支本身存在业务层和 UI 层大量差异，因此即便公共层能顺滑接入，也不能等同于“整个分支升级绝对无冲突”。

### 实际判断

如果下次 `main` 更新主要集中在以下范围：

1. OpenClaw runtime 版本
2. 公共插件版本
3. `openclaw.json` 的公共配置结构
4. 插件安装/加载脚本

那么接入应当会比这次顺滑很多，通常只需要：

1. 同步 `package.json` 的 `openclaw` 区块
2. 重新执行插件安装
3. 检查 `openclaw.json`
4. 再做一次启动验证

如果下次 `main` 更新包含新的上游 breaking change，则仍需要人工处理，但不会再像这次一样同时踩中：

1. 客户端入口变化
2. 配置 schema 变化
3. 目录结构变化
4. ClawHub 授权缺失
5. 历史残留插件污染

换句话说：

1. 这次已经把“基础设施层的坑”大部分填平了。
2. 下次继续接 `main`，大概率是“正常升级工作”，而不是“多点爆炸式排障”。

## 建议的下次接入流程

建议后续每次跟进 `main` 时按下面顺序执行：

1. 先对比 `main:package.json` 中的 `openclaw.version` 和 `openclaw.plugins`。
2. 确认 `~/Library/Application Support/clawhub/config.json` 仍存在且 token 有效。
3. 执行 `node scripts/ensure-openclaw-plugins.cjs`。
4. 检查 `vendor/openclaw-runtime/current/third-party-extensions/` 是否齐全。
5. 确认 `vendor/openclaw-runtime/current/extensions/` 中没有历史残留第三方插件。
6. 执行 `npx tsc -p electron-tsconfig.json --noEmit`。
7. 执行 `npm run electron:dev`，检查：
   - gateway 是否 `ready`
   - `GatewayClient` 是否握手成功
   - `openclaw-lark` 是否注册命令
   - 对应渠道是否正常启动

这样做能把风险收敛在最小范围内，符合：

1. `KISS`：先验证公共底座，再碰业务层。
2. `DRY`：把重复踩过的坑固化成固定检查流程。
3. `YAGNI`：只对这次已经暴露过的关键风险点做必要检查，不做过度预案。
