# 青数覆盖层：品牌元数据梳理

## 1. 背景与目标

这份文档用于把当前 `front-design-merge` 分支里与“青数品牌”直接相关、且后续最容易在继续拉齐 `main` 时被冲散的内容单独提炼出来。

推荐采用的最稳方案是：

1. 先把品牌层、内置治理链、UI/样式、唤醒、TTS 等“覆盖层”文档化。
2. 后续从最新 `main` 新拉分支。
3. 再按文档把这些覆盖层有选择地回贴到新分支。

这样做的原因：

1. 符合 `KISS`：先把边界梳理清楚，再迁移，避免在脏分支上继续做高耦合大合并。
2. 符合 `YAGNI`：当前先只提炼“已经明确需要保留”的品牌层，不预先重构全部品牌系统。
3. 符合 `SOLID`：把“公共主干”和“品牌覆盖层”分开识别，后续迁移更容易分模块执行。
4. 符合 `DRY`：后续每次从 `main` 接入时，直接复用这份清单，不再重新人工找落点。

## 2. 核心结论

当前分支的品牌体系不是单一命名，而是三层并存：

1. 外显产品品牌：`QingShuClaw`
2. 平台与业务语义：`灵工打卡`、`青数`、`青数CLAW`
3. 底层遗留技术标识：`lobsterai`、`LobsterAI`

当前分支的品牌视觉也不是单一来源，而是两层并存：

1. 全局主题 token：`--lobster-primary` 及其相关 CSS 变量
2. 局部品牌强调：大量直接写在组件里的 `emerald/teal` Tailwind 类

因此，后续从 `main` 新起分支时，不能只替换几个文案或一张 logo。最少需要同时回贴：

1. 外显名称与品牌介绍
2. logo / 图标 / 品牌徽记
3. 品牌色 token 与局部品牌强化样式
4. 主工作台中 `main agent / managed agent` 的品牌映射

## 4. 品牌名体系

### 4.1 外显产品名

当前对用户可见的产品名主线是 `QingShuClaw`。

核心来源：

1. `src/main/appConstants.ts`
   - `APP_NAME = 'QingShuClaw'`
2. `electron-builder.json`
   - `productName = QingShuClaw`
   - `executableName = QingShuClaw`
   - 协议展示名 `name = QingShuClaw`
3. `src/renderer/components/Settings.tsx`
   - 关于页直接展示 `APP_NAME`

主要生效位置：

1. Electron 应用名称与窗口标题
2. 安装包产物名称
3. 系统协议展示名
4. 关于页产品标题
5. 托盘 tooltip 与部分系统级文案

### 4.2 平台与业务语义名

当前 UI 中承接“业务语义”的不是 `QingShuClaw` 本身，而是：

1. `灵工打卡`
2. `青数`
3. `青数CLAW`

核心来源：

1. `src/renderer/services/i18n.ts`
   - `workbenchBrandLine1`
   - `workbenchBrandLine2`
   - `workbenchConversationGroupMain`
   - `aboutBrandEyebrow`
   - `aboutBrandDescription`
   - `authLoginWelcomeSubtitle`
   - `productBrandSignature`
   - `privacyDialogTitle / privacyDialogDesc / privacyDialogLinkText`
2. `src/renderer/services/brandRuntime.ts`
   - 默认协议标题、描述、链接文案

主要生效位置：

1. 一级侧栏底部品牌签名
2. 二级侧栏 `main agent` 命名与分组
3. 登录欢迎区
4. 关于页品牌介绍区
5. 协议弹窗与更新相关品牌承接文案

### 4.3 底层遗留技术标识

当前仓库中仍存在大量 `lobsterai` / `LobsterAI` 标识。这些内容很多已经不只是“品牌词”，而是兼容性标识，不能在没有迁移方案时直接重命名。

主要来源：

1. `src/main/appConstants.ts`
   - `APP_USER_DATA_DIR_NAME = 'LobsterAI'`
   - `APP_ID = 'lobsterai'`
   - `DB_FILENAME = 'lobsterai.sqlite'`
2. `package.json`
   - `name = lobsterai`
   - `author.name = LobsterAI`
3. `electron-builder.json`
   - `appId = com.lobsterai.app`
   - `protocols.schemes = ['lobsterai']`
4. `src/renderer/services/endpoints.ts`
   - 更新、下载、Portal、Skill 商店、登录 URL 仍然走 `lobsterai` 路径
5. 任务 / 会话 / 测试夹具中仍有 `agent:*:lobsterai:*`

这层内容当前建议：

1. 不把它视为纯视觉品牌层。
2. 在新分支回贴时优先保留兼容，不做激进重命名。
3. 如果未来确实要统一重命名，必须单独做“协议 / 用户目录 / 数据库 / 更新链路 / session key”迁移方案。

## 5. 品牌介绍与协议文案体系

### 5.1 运行时品牌介绍主来源

品牌介绍的运行时主来源是 `src/renderer/services/brandRuntime.ts`。

当前默认配置包含：

1. 协议标题：`灵工打卡 QingShuClaw 使用协议`
2. 协议描述：`QingShuClaw 搭载于青数中台，仅在用户权限范围内访问灵工打卡数据`

1. 协议弹窗的直接数据源

## 7. 品牌色体系

### 7.1 全局主题色 token

全局品牌主色当前主要由 `src/renderer/theme/css/themes.css` 提供。

默认主题下的关键值：

1. `--lobster-primary: #10b981`
2. `--lobster-primary-hover: #059669` 或 `#34d399`
3. `--lobster-primary-muted: rgba(16,185,129,0.12)`

这些 token 会通过 `src/renderer/index.css` 和组件内 `bg-primary / text-primary / border-primary` 等消费位扩散到整个工作台。

### 7.2 局部直接写死的品牌色

除了全局 token 之外，当前品牌强化还依赖不少组件里的直接颜色类，例如：

1. `from-emerald-400`
2. `via-emerald-500`
3. `to-teal-500`
4. `text-emerald-600`
5. `dark:text-emerald-300`

高频位置：

1. `src/renderer/components/branding/QingShuBrandMark.tsx`
2. `src/renderer/components/layout/PrimarySidebar.tsx`
3. `src/renderer/components/layout/SecondarySidebar.tsx`
4. `src/renderer/components/LoginButton.tsx`
5. `src/renderer/components/WakeActivationOverlay.tsx`

### 7.3 当前品牌色的结构性问题

当前品牌色体系可用，但存在一个后续迁移风险：

1. 一部分颜色走 token
2. 一部分颜色直接写死在组件

这意味着后续如果只是把 `themes.css` 回贴过去，而漏掉品牌组件里的 `emerald/teal` 类，最终视觉会只“像了一半”。

因此从 `main` 新分支回贴时，品牌色不能只迁 `themes.css`，还要一起回贴品牌组件和侧栏局部样式。

## 9. 从 main 新分支回贴时的最稳迁移建议

### 9.1 推荐顺序

1. 先从最新 `main` 新拉分支。
2. 第一层只回贴“外显品牌元数据”，不要先碰底层 `lobsterai` 标识迁移。
3. 第二层回贴“品牌运行时文案”与“关于页 / 协议页落点”。
4. 第三层回贴 `QingShuBrandMark + PrimarySidebar + SecondarySidebar + LoginButton`。
5. 第四层回贴 `themes.css` 中当前默认主题的青绿色主色，以及必要的局部 `emerald/teal` 强化样式。
6. 最后再做联调验收。

### 9.2 最应优先保留的文件

高优先级：

1. `src/renderer/components/branding/QingShuBrandMark.tsx`
2. `src/renderer/components/layout/PrimarySidebar.tsx`
3. `src/renderer/components/layout/SecondarySidebar.tsx`
4. `src/renderer/components/LoginButton.tsx`
5. `src/renderer/services/brandRuntime.ts`
6. `src/renderer/services/i18n.ts`
7. `src/renderer/components/Settings.tsx`
8. `src/renderer/theme/css/themes.css`

中优先级：

1. `public/logo.png`
2. `build/icons/**`
3. `resources/tray/**`
4. `scripts/generate-app-icon.js`
5. `scripts/generate-tray-icons.js`

低优先级但需谨慎评估：

1. `src/main/appConstants.ts`
2. `package.json`
3. `electron-builder.json`
4. `src/renderer/services/endpoints.ts`

低优先级不是因为不重要，而是因为这些文件带有较强兼容性和发布链路含义，迁移时要谨慎。

### 9.3 验收清单

从 `main` 新分支回贴后，至少要检查这些点：

1. 启动后的窗口名、Dock 名、关于页标题是否为 `QingShuClaw`
2. 一级侧栏底部是否仍显示“灵工打卡 / 青数”品牌签名
3. 二级侧栏 `main` Agent 是否仍映射为 `青数CLAW`
4. 青数内置 Agent 是否仍有统一品牌徽记
5. 登录区是否仍保留青数品牌副标题和角标
6. 关于页是否仍显示品牌介绍、平台 badge、logo
7. 协议弹窗是否仍为“灵工打卡 QingShuClaw 使用协议”
8. 更新弹窗是否仍能正确读取品牌运行时文案
9. 默认主题主色是否仍是青绿色，而不是被 `main` 的默认色覆盖

## 10. 风险与注意事项

### 10.1 不要把所有 `lobsterai` 当成可直接替换的品牌文案

这些内容很多已经带有兼容语义：

1. 协议 scheme
2. 用户数据目录
3. 数据库文件名
4. 更新与 Portal 地址
5. session key / task key

如果直接整体替换，可能带来：

1. deep link 失效
2. 用户旧数据无法继承
3. 自动更新链断开
4. 定时任务或会话关联失配

### 10.2 品牌色不能只迁 token

如果只迁 `themes.css`，但漏掉 `QingShuBrandMark` 与侧栏里的 `emerald/teal` 类：

1. 主按钮会是青绿色
2. 但品牌徽记、品牌签名、主 Agent 头像未必一致

最终会出现“色调接近，但品牌识别不完整”的半成品状态。

### 10.3 关于页与协议页是最容易漏的

后续继续从 `main` 拉齐时，最容易只关注主工作台，而漏掉：

1. 关于页品牌介绍
2. 协议弹窗
3. 更新弹窗的品牌承接文案
4. 登录欢迎区

这四块都属于用户第一眼可感知的品牌位，不能省。
