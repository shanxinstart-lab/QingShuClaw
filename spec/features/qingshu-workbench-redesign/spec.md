# Feature Specification: QingShuClaw 工作台展示与交互改版

**Feature ID**: qingshu-workbench-redesign
**Created**: 2026-04-17
**Status**: Draft

## 问题陈述

当前 QingShuClaw 主界面采用“综合侧栏 + 欢迎区 + 输入框”的首页结构，已经能够支持新建任务、继续会话、选择模型、挂载技能、上传附件等核心能力，但存在以下问题：

1. 一级导航、Agent 切换、历史会话、设置入口都挤在同一栏，职责过重。
2. 首页空态更偏展示页，不像一个可持续使用的桌面工作台。
3. 对话列表、Agent 分组、历史展开、任务视图、技能视图、应用视图之间缺乏统一的信息架构。
4. 当前交互路径没有形成“主导航 -> 模块侧栏 -> 主工作区”的稳定心智。

本次改版目标不是新增业务功能，而是按新版本样式重构工作台的信息架构、模块分区、展示层次与核心交互路径。
现有业务能力原则上继续沿用，本次仅对界面组织方式、入口位置、展示规则和局部状态编排进行调整。

## 改版目标

### 目标 1：建立稳定的三栏工作台结构

主界面升级为：

- 一级侧边栏
- 二级侧边栏
- 系统主工作区

### 目标 2：对齐新的模块导航模型

一级主导航固定为：

- 对话
- 任务
- 技能
- 应用

### 目标 3：让“对话”模块成为完整工作区

在“对话”模块下，用户应能完成：

- 搜索对话或 Agent
- 创建专属 Agent
- 在指定分组下创建新对话
- 浏览最新对话
- 展开完整历史记录
- 在主区持续进行对话

### 目标 4：为任务、技能、应用建立清晰独立视图

除“对话”外，其余模块也必须拥有明确的主工作区定义，而不是沿用旧版混合式页面。

### 目标 5：在不丢失现有核心能力的前提下完成改版

模型切换、附件上传、语音输入、插件挂载、Markdown 渲染、任务历史等现有能力需保留，不因改版丢失。
本次工作以展示重组和交互调整为主，不引入新的核心业务能力。

## 非目标

以下内容不在本次 PRD 范围内：

- 重做底层会话存储模型
- 重做消息协议或插件协议
- 本轮新增新的一级业务模块
- 本轮新增复杂权限系统
- 本轮重做设置页信息结构
- 本轮进行移动端适配
- 本轮进行完整深色主题重设计
- 为适配新样式而额外引入新的业务实体、独立存储模型或新的任务协议

## 用户场景

### 场景 1：从“青数CLAW”分组创建新对话

**Given** 用户进入“对话”模块  
**When** 点击二级侧边栏中的“青数CLAW”分组标题文字  
**Then** 系统应直接创建并进入一个新对话  
**And** 主工作区展示新的对话视图

### 场景 2：展开某个分组查看最近会话

**Given** 用户位于“对话”模块  
**When** 点击分组标题右侧箭头  
**Then** 展开或收起该分组的最近对话记录  
**And** 展开后最多默认显示 5 条最近会话

### 场景 3：查看完整历史会话

**Given** 某分组下对话记录超过 5 条  
**When** 用户点击“展开更多...”  
**Then** 主内容区右侧应推出历史记录侧边栏  
**And** 用户可以搜索和查看该分组下的完整历史对话

### 场景 4：在主工作区继续对话

**Given** 用户已经进入某个对话  
**When** 用户在底部输入框输入内容并发送  
**Then** 主区应继续渲染对话流  
**And** 保留模型切换、语音、附件、挂载插件等入口

### 场景 5：切换到“任务”模块

**Given** 用户点击一级侧边栏中的“任务”  
**When** 进入任务模块  
**Then** 主工作区展示“当前任务列表 / 历史任务记录”两个 Tab  
**And** 用户可以清晰区分运行中任务与已完成任务

### 场景 6：切换到“技能”模块

**Given** 用户点击一级侧边栏中的“技能”  
**When** 进入技能模块  
**Then** 主工作区展示技能管理页  
**And** 用户可以搜索技能、查看技能卡片并新增技能

### 场景 7：切换到“应用”模块

**Given** 用户点击一级侧边栏中的“应用”  
**When** 进入应用模块  
**Then** 主工作区展示应用列表  
**And** 用户可以查看应用连接状态并执行“连接”操作

## 产品原则

### KISS

- 一级导航只保留四个核心模块。
- 二级侧边栏只承载当前模块最关键的对象列表与操作。
- 主工作区优先保证正在进行的工作内容，不再保留大面积欢迎展示。

### YAGNI

- 本轮只对齐已明确提出的导航、分组、主视图区和模块页面规格。
- 不提前加入未被需求明确提出的新模块、新标签或新侧边栏。

### SOLID

- 一级侧边栏负责主模块切换。
- 二级侧边栏负责当前模块对象导航与筛选。
- 主工作区负责内容呈现与核心操作。

### DRY

- 最大化复用现有会话渲染、输入框、模型切换与列表数据能力。
- 避免为新样式重造一套会话、任务或技能的数据流。

## 信息架构

### 一级侧边栏（主导航）

一级侧边栏固定展示在最左侧，包含三个区域：

#### 顶部：用户信息

- 显示用户头像
- 显示用户名 `admin`

#### 中部：核心导航

按顺序固定展示 4 个一级菜单：

1. 对话
2. 任务
3. 技能
4. 应用

#### 底部：系统与版本

- 上方包含一个“设置”入口
- 最底部展示两行文本标识：
  - `灵工打卡`
  - `青数`

### 二级侧边栏

二级侧边栏在不同一级模块下展示对应内容。

在“对话”模块下，二级侧边栏包含：

#### 顶部操作区

- 一个全局搜索框
- 搜索框提示词为：`搜索对话或Agent...`
- 搜索框下方为独立按钮：`+ 创建专属 Agent`

#### 对话分组目录

包含两个第一层级分组，且两者为平行关系：

- `青数CLAW`
- `内置 Agent - 翻译助手`

#### 分组交互逻辑

- 点击分组标题文字本身：直接创建并进入一个新对话
- 点击分组标题右侧箭头：展开或收起该分组
- 每个分组展开后，默认仅展示最新最多 5 条对话记录
- 若该分组对话记录超过 5 条，则列表底部显示 `展开更多...`

### 系统主工作区

系统主工作区根据一级导航切换内容。

#### 1. 对话主内容区

##### 主对话视图

顶部区域包含：

- 当前对话标题
- 当前挂载的 Agent / 插件名称
- `更多` 操作入口

中部区域为对话流内容区，需支持：

- 文本对话渲染
- 插件调用记录渲染
- Markdown 内容渲染
- Markdown 数据表格渲染

底部区域为悬浮对话输入框，包含：

- 模型切换下拉菜单
- 语音入口图标
- 上传附件入口图标
- 挂载插件入口图标
- 发送按钮

##### 历史记录面板

当用户点击 `展开更多...` 后，从主工作区右侧推出历史记录面板。

该面板包含：

- 顶部标题：`历史对话(所属分类名)`
- 一个关闭按钮
- 一个专属搜索框，用于对当前分类历史对话做模糊搜索过滤
- 完整历史对话列表

每条历史对话记录需至少展示：

- 对话标题
- 产生时间

#### 2. 任务主内容区

任务模块顶部展示两个并排 Tab：

- `当前任务列表`
- `历史任务记录`

右侧展示这两类列表下的总任务数量。

##### 当前任务列表

展示所有正在运行中的任务卡片。

每张卡片需包含：

- 任务名称
- 定时执行规则
- `运行中` 状态标识

##### 历史任务记录

展示所有已结束的任务卡片。

每张卡片需包含：

- 任务名称
- 触发时间
- `已完成` 状态标识

#### 3. 技能主内容区

技能模块顶部区域包含：

- 左侧标题：`技能管理`
- 右侧按钮：`+ 添加新技能`

页面上方提供一个全局技能搜索框。

下方为网格布局的技能列表卡片。

每张技能卡片需包含：

- 技能名称
- 一段功能用途描述

#### 4. 应用主内容区

应用模块顶部区域包含：

- 左侧标题：`应用列表`
- 右侧统计文字，例如：`已连接: X / Y`

下方为网格布局的应用列表卡片。

每张应用卡片需包含：

- 应用 Logo 图标
- 应用名称
- 简介说明
- 右侧状态操作按钮

状态按钮规则：

- 未连接时显示：`连接`
- 成功连接后显示：`已连接`

## 交互与展示需求

以下需求默认建立在“现有能力已存在”的前提上，重点描述新样式下的界面结构、展示规则、入口编排与轻状态逻辑调整，不单独定义新的业务能力。

### FR-1：一级侧边栏结构必须固定

- **FR-1.1**：一级侧边栏必须包含顶部用户信息、中部核心导航、底部系统与版本三个区域。
- **FR-1.2**：一级导航菜单必须严格按 `对话 / 任务 / 技能 / 应用` 顺序展示。
- **FR-1.3**：底部必须提供“设置”入口和两行品牌文案 `灵工打卡`、`青数`。

### FR-2：对话模块二级侧边栏必须支持常驻搜索与 Agent 创建

- **FR-2.1**：二级侧边栏顶部必须提供搜索框，提示词为 `搜索对话或Agent...`。
- **FR-2.2**：搜索框下方必须提供独立按钮 `+ 创建专属 Agent`。
- **FR-2.3**：搜索需支持至少对对话标题与 Agent 名称做模糊过滤。

### FR-3：对话分组目录必须支持两类一级分组

- **FR-3.1**：对话模块二级侧边栏必须包含 `青数CLAW` 与 `内置 Agent - 翻译助手` 两个平行分组。
- **FR-3.2**：分组标题文字点击行为为“新建并进入对话”，不得与“展开/收起”复用为同一点击区域。
- **FR-3.3**：分组标题右侧箭头为唯一的展开/收起入口。

### FR-4：分组下的历史记录展示必须支持截断与延展

- **FR-4.1**：每个分组展开后默认最多展示 5 条最近对话记录。
- **FR-4.2**：若记录数大于 5，则列表底部显示 `展开更多...`。
- **FR-4.3**：点击 `展开更多...` 后，主工作区右侧推出详细历史记录侧边栏。

### FR-5：详细历史记录侧边栏必须支持独立浏览

- **FR-5.1**：历史记录侧边栏顶部必须展示当前所属分类标题。
- **FR-5.2**：历史记录侧边栏必须提供关闭按钮。
- **FR-5.3**：历史记录侧边栏内必须提供专属搜索框。
- **FR-5.4**：历史记录列表必须展示完整对话记录，且每条至少包含标题与产生时间。

### FR-6：对话主工作区必须支持稳定的对话壳结构

- **FR-6.1**：对话主工作区顶部必须展示当前对话标题。
- **FR-6.2**：顶部右侧必须展示当前挂载的 Agent / 插件名称以及“更多”操作入口。
- **FR-6.3**：主内容区必须支持文本对话、插件调用记录、Markdown 与 Markdown 表格渲染。
- **FR-6.4**：底部输入区必须以悬浮形式固定在主工作区底部。

### FR-7：对话输入区必须保留关键能力入口

- **FR-7.1**：输入区必须包含模型切换下拉菜单。
- **FR-7.2**：输入区必须包含语音入口。
- **FR-7.3**：输入区必须包含上传附件入口。
- **FR-7.4**：输入区必须包含挂载插件入口。
- **FR-7.5**：输入区必须包含发送按钮。

### FR-8：任务模块必须具备双视图 Tab

- **FR-8.1**：任务模块必须展示 `当前任务列表` 与 `历史任务记录` 两个 Tab。
- **FR-8.2**：任务模块右侧必须展示总任务数量信息。
- **FR-8.3**：当前任务卡片必须展示任务名称、定时执行规则与 `运行中` 状态。
- **FR-8.4**：历史任务卡片必须展示任务名称、触发时间与 `已完成` 状态。

### FR-9：技能模块必须具备管理页结构

- **FR-9.1**：技能模块顶部必须展示标题 `技能管理`。
- **FR-9.2**：技能模块顶部右侧必须提供 `+ 添加新技能` 按钮。
- **FR-9.3**：技能模块必须提供全局技能搜索框。
- **FR-9.4**：技能模块必须以网格卡片方式展示技能名称与功能描述。

### FR-10：应用模块必须具备连接状态页结构

- **FR-10.1**：应用模块顶部必须展示标题 `应用列表`。
- **FR-10.2**：应用模块顶部右侧必须展示连接统计信息，如 `已连接: X / Y`。
- **FR-10.3**：应用卡片必须包含 Logo、名称、简介与状态按钮。
- **FR-10.4**：未连接状态按钮展示 `连接`，已连接状态按钮展示 `已连接`。

### FR-11：现有核心能力必须保留

- **FR-11.1**：现有会话数据、历史记录与状态信息不得因改版丢失。
- **FR-11.2**：模型切换、附件上传、语音输入能力必须继续可用。
- **FR-11.3**：定时任务、技能管理、应用连接入口必须能从新工作台结构进入。

### FR-12：本次改版默认不新增核心业务能力

- **FR-12.1**：若现有能力已满足需求，本次仅允许调整展示位置、交互方式和状态呈现。
- **FR-12.2**：因新样式产生的逻辑调整，应优先归类为界面状态逻辑，而非新增业务逻辑。
- **FR-12.3**：任何超出本 PRD 的新业务能力，如新协议、新实体、新流程，需单独评审后再进入范围。

## 视觉与体验要求

### 视觉方向

新版本应体现：

- 清晰的三栏结构
- 浅色、轻量、桌面工作台风格
- 明确的主次层级
- 对话主区更强的连续阅读感

### 版式要求

- 一级侧边栏为窄栏，优先承担导航身份
- 二级侧边栏宽度稳定，承担搜索、分组与列表浏览
- 主工作区优先保证消息流与输入区的完整显示
- 历史记录面板从右侧推出，不覆盖整个主工作区

### 状态反馈

至少需明确反馈以下状态：

- 一级导航选中
- 分组展开/收起
- 当前对话选中
- 搜索无结果
- 任务运行中
- 任务已完成
- 应用已连接 / 未连接

## 文案要求

- 一级导航、分组标题、按钮、Tab、状态词均使用中文。
- 新增文案必须接入 i18n，并补齐中英文翻译。
- 业务文案优先采用操作型表达，避免宣传式空态文案占据主视区。

## 技术实现边界

### 可复用部分

以下能力应优先复用：

- 现有会话列表数据源
- 现有消息渲染组件
- 现有 Markdown 渲染能力
- 现有输入框提交逻辑
- 现有模型切换逻辑
- 现有任务、技能、Agent、MCP 相关数据流

### 实现原则

- 能复用现有接口与状态结构的，不新增中间层。
- 能通过视图拆分与组件重组实现的，不新增新的业务抽象。
- 能通过展示规则和前端局部状态完成的，不修改底层持久化模型。

### 建议新增组件

建议新增以下结构组件：

- `src/renderer/components/layout/PrimarySidebar.tsx`
- `src/renderer/components/layout/SecondarySidebar.tsx`
- `src/renderer/components/layout/WorkbenchShell.tsx`
- `src/renderer/components/cowork/ConversationWorkspaceHeader.tsx`
- `src/renderer/components/cowork/ConversationHistoryDrawer.tsx`
- `src/renderer/components/tasks/TaskWorkspaceView.tsx`
- `src/renderer/components/skills/SkillWorkspaceView.tsx`
- `src/renderer/components/apps/AppWorkspaceView.tsx`

### 技术约束

- 优先采用渐进式迁移，避免一次性推翻现有业务实现。
- 布局与交互重构优先，底层状态结构改动保持最小化。
- 历史记录侧边栏应优先基于现有会话数据实现，不新建独立存储。

## 分阶段实施规划

### 阶段 1：搭建一级侧边栏、二级侧边栏和主工作区骨架

交付内容：

- 一级侧边栏基础布局
- 二级侧边栏基础布局
- 主工作区容器
- 旧版 Sidebar 的职责拆分

验收标准：

- 主界面形成稳定三栏结构
- 一级导航可切换到 `对话 / 任务 / 技能 / 应用`

### 阶段 2：完成“对话”模块改造

交付内容：

- 常驻搜索框
- `+ 创建专属 Agent` 按钮
- 两个对话分组
- 默认 5 条历史记录与 `展开更多...`
- 主对话区顶部结构
- 右侧历史记录面板
- 底部悬浮输入框重排

## 成功标准

| 指标 | 目标 |
|------|------|
| 一级导航可辨识度 | 用户可在 3 秒内理解 4 个主模块 |
| 对话入口效率 | 用户可在二级侧边栏 1 次点击内创建新对话 |
| 历史记录可达性 | 默认看到最近 5 条，完整历史可通过右侧面板访问 |

| 模块完整性 | 任务、技能、应用均具备独立且清晰的主工作区 |

## 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| 旧版 Sidebar 状态与新三栏结构耦合较深 | 高 | 中 | 先拆布局壳，再逐步迁移逻辑 |
| 分组标题点击与展开箭头交互容易混淆 | 中 | 高 | 将标题点击区和箭头点击区明确分离 |
| 历史记录面板推出后压缩主工作区宽度 | 中 | 中 | 控制抽屉宽度，保证主区阅读宽度 |
| 任务、技能、应用视图重构过程中出现功能回归 | 中 | 中 | 分模块迁移并逐步验收 |

## 待确认事项

1. “青数CLAW”与“内置 Agent - 翻译助手”是否为首版固定写死分组，还是后续可配置。
2. “创建专属 Agent”是否直接进入 Agent 创建流，还是打开弹窗表单。
3. 历史记录侧边栏推出方式是覆盖式抽屉还是挤压式推开，首版是否必须严格按“推开”实现。
4. 应用模块中的“已连接: X / Y”统计口径是否仅统计当前支持接入的应用。
5. 当前对话顶部展示的“挂载 Agent / 插件名称”是否需要支持多个标签并列显示。

## 验收建议

建议按以下顺序验收：

1. 信息架构验收
   确认一级导航、二级侧边栏与主工作区的关系是否符合本 PRD。

2. 交互验收
   确认分组点击、展开、历史抽屉、输入区与模块切换是否符合需求。

3. 视觉验收
   确认三栏比例、列表样式、主区头部、输入区与状态表现是否达到目标样式。
